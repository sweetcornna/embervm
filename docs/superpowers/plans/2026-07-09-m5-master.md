# EmberVM M5 (Agent-native fork/branch/rollback) — Master Spec

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans — phases execute
> inline, each phase: implement (TDD portable surfaces) → local checks → push → CI green
> before the next. This file locks cross-phase decisions so phase work cannot drift.
> Predecessors: ADR-0003 (chunked restore), ADR-0004 (tiers), ADR-0005 (M4 cluster).

**Goal:** checkpoint/fork/rollback as first-class sandbox API (docs/zh/03 §3 M5, 04 #4/#8):
one sandbox forks 10 branches that execute in parallel while the parent keeps running; any
checkpoint can be forked from or rolled back to; per-exec-step auto-checkpoints give
time-travel replay. Ships as `v0.5.0-m5`.

**Architecture:** A checkpoint IS what every pause already produces — a memory layer p<N>
(chunked, content-addressed) plus the ZFS snapshot @p<N> taken by the pause path — surfaced
as a named, PG-recorded object. Fork generalizes M4's golden fast-create: `zfs clone
parent@p<N>` + copy the parent's manifest chain (p1..pN) + snapfile-pN + ws.json into a new
identity + ordinary chunked hot resume (~150-250ms CI). Memory is CoW by construction: chunks
are content-addressed and shared, children fault pages from the same local store. Rollback is
the DeltaBox "layer switch": `zfs rollback -r @p<N>` + trim the in-memory chain + hot resume
from p<N> — resume and rollback share one mechanism (04 #8). Entropy: every fork/rollback
passes LoadSnapshot (VMGenID reseeds the guest PRNG) + the guestd /resumed hook (charter risk
#4: fork is explicit, snapshot-resume-once stays the default elsewhere).

**Tech Stack:** existing pkg tree (nodeagent/nodeapi/controlplane), ZFS clone/rollback,
PostgreSQL migration 0005, chunked uffd restore. No new dependencies.

## Global Constraints

- Charter (docs/zh/03 §3 M5, verbatim): fork/branch/rollback 一等 API(《04》#4/#8):uffd
  CoW 内存克隆 + ZFS 层链,checkpoint/rollback 目标 <100ms; time-travel 调试:每步自动快照 +
  任意步 fork 重放; 退出标准:单沙箱 fork 出 10 分支并行执行,父实例不停顿.
- The <100ms checkpoint/rollback figure is a bare-metal target (DeltaBox: checkpoint 14ms /
  rollback 5ms). CI latencies are relative (ADR-0001): gates measure and log; the hard CI
  gate is correctness + the 10-branch parallel exit criterion, with generous wall-clock caps.
- All standing contracts hold: monotone seq/tags (NEVER reset snapCount — rollback continues
  the sequence), producer-is-truth JSON, chunked+jailed required for clone-restore paths,
  GUID lineage (children clone the parent's REAL snapshot), M0-M4 suites stay green.
- Fork/rollback require RestoreMode=chunked (the M2+ pipeline; raw-memfile M1 paths are not
  eligible — same rule as the jailer).
- Fork is SAME-NODE in M5 (children placed on the parent's node; the parent's staging chain
  is node-local). Cross-node fork = resume-the-child-from-L1 later; recorded in ADR-0006.

## Design decisions (ADR-0006 material)

**D1 — Checkpoint = the pause artifact, named and recorded.** `POST
/v0/sandboxes/{id}/checkpoints {"tag": "step-3"}` (tag optional; default `cp-<seq>`) drives
the existing SnapshotSandbox verb (pause → diff layer p<N> + zfs @p<N> → resume; the pause
window IS the checkpoint cost — diff layers keep it small). The control plane parses the
layer from the verb's return (`<id>@<tag>-<N>`, producer-defined format) and inserts into a
new `checkpoints` table `(sandbox_id, tag, layer, created_at, PRIMARY KEY(sandbox_id, tag))`.
`GET /v0/sandboxes/{id}/checkpoints` lists them. Duplicate tag → 409. Checkpoints are valid
while the sandbox's chain stays local (HOT); tiered-out parents refuse fork/rollback with 409
(resume first) — lineage-aware tiering is out of scope (D5 guards).

**D2 — Fork = golden fast-create, generalized.** Node verb `Fork(ctx, parentID, layer,
newID)`: `CloneSandboxFrom(newID, parentID, "p<N>")` (exists since M4 golden), copy
layer-p1.json..layer-p<N>.json + snapfile-p<N> + ws.json from the parent's snap dir into the
child's, seed `sb.layers` from the manifests, `diskOrigin = {parentID, p<N>}` (existing
descriptor machinery — the child's pause/L1 write-through and cross-node restore then work
unchanged), fresh netns lease, egress inherited from the parent, chunked hot resume. The
parent is NOT paused by fork (its chain files are read-only after their pause). REST: `POST
/v0/sandboxes/{id}/fork {"checkpoint": "step-3"}` — omitted checkpoint auto-creates one
first (the live-parent "branch now" UX; one checkpoint then N forks reuse it by tag). Child
is a full sandbox row: owner-scoped, quota-counted (CountActiveSandboxes), `parent_id` +
`forked_from` columns set, placed on the parent's node explicitly (no scheduler Place call).

**D3 — Rollback = layer switch on the SAME identity.** Node verb `Rollback(ctx, id, layer)`:
CAS RUNNING→PAUSING fails callers racing verbs; kill FC + uffd, `zfs rollback -r <ds>@p<N>`
(dataset path unchanged — no jail rebind, no mountpoint churn), trim `sb.layers` to ..p<N>,
`snapLayer = "p<N>"`, delete layer-p(>N).json + snapfile-p(>N) staging, hot resume from p<N>.
`snapCount` keeps counting upward (monotone tags across the rollback). Semantics, documented
loudly: rollback DISCARDS checkpoints newer than the target (their zfs snapshots die with
`rollback -r`); it is REFUSED (409) while a newer checkpoint has live fork children (zfs
would refuse the snapshot destroy anyway — we surface it before touching anything). The
control plane deletes the discarded `checkpoints` rows in the same transaction that records
the rollback. Time-travel replay therefore uses FORK-from-step (non-destructive), rollback is
the destructive "go back" (DeltaBox switch-layer).

**D4 — Time-travel = exec-with-checkpoint.** The exec body gains `"checkpoint": true`: the
handler checkpoints BEFORE running the command (tag `step-<seq>`), so a step's checkpoint is
the state the command saw — fork from `step-k` replays step k. The exec response gains
`"checkpoint": "<tag>"` alongside the guest fields (additive JSON). No engine, no history
subsystem: the checkpoints table IS the timeline (04 #4 每步自动快照 + 任意步 fork 重放).

**D5 — Lineage guards (ZFS clone dependencies made loud).** `sandboxes.parent_id text`,
`sandboxes.forked_from text` (checkpoint tag). A parent with live (non-terminal) children:
DELETE → 409 listing the children; the lifecycle engine's TTL scans SKIP it (ReleaseLocal
would `zfs destroy` a dataset with dependent clones — refused at the ZFS layer, so we refuse
politely first). Children tier/destroy freely (their datasets are the clones). Killing a
child never touches the parent. This is the M5-minimal rule; promote/lineage-GC is future
work (carried debt).

**D6 — Entropy & identity per branch.** Every fork/rollback goes through LoadSnapshot →
VMGenID reseed → guestd /resumed hook (existing). Each child gets its own netns lease (same
guest IP by design, isolation by namespace — M0 contract). Charter risk #4 satisfied: fork
is the explicit multi-resume declaration; nothing else resumes one snapshot twice.

**D7 — API shapes (all owner-scoped, quota where sandboxes are created):**

```
POST /v0/sandboxes/{id}/checkpoints {"tag"?}     → 201 {"tag","layer","created_at"}
GET  /v0/sandboxes/{id}/checkpoints              → 200 [{"tag","layer","created_at"}...]
POST /v0/sandboxes/{id}/fork {"checkpoint"?}     → 201 Sandbox (child; parent_id/forked_from set)
POST /v0/sandboxes/{id}/rollback {"checkpoint"}  → 200 Sandbox
POST /v0/sandboxes/{id}/exec {..., "checkpoint": true} → 200 {...exec fields, "checkpoint": "step-N"}
```

nodeapi additions (Agent interface + HTTP transport + mocks): `Fork(ctx, parentID, layer,
newID string) (SandboxStatus, error)`, `Rollback(ctx, sandboxID, layer string)
(SandboxStatus, error)`. Layer strings are the `p<N>` names.

**D8 — Exit gates (e2e-m5, `--- PASS:` guards).**
- `TestFork10Branches` (THE exit gate, 退出标准 verbatim): parent runs a background counter
  (`sh -c 'while true; do date +%s%N >> /count; done'` style via exec), one checkpoint, 10
  parallel forks; every child RUNNING + exec-verified + sees the checkpoint marker; children
  diverge (each writes its own file; parent and siblings never see it); the parent's counter
  file KEEPS GROWING across the fork window (父实例不停顿 — sampled before/after, strictly
  increasing) and the parent was never observed outside RUNNING.
- `TestRollbackCheckpoint`: marker A → checkpoint → marker B → rollback → A present, B gone,
  guest seq continuity (no reboot), re-checkpoint after rollback works (monotone tags), and
  rollback-with-live-forks-off-newer-checkpoint → 409.
- `TestTimeTravelReplay`: 3 exec steps with `"checkpoint": true` mutating a file; fork from
  `step-2`'s tag → child sees exactly the pre-step-2 state; replaying step 2 in the child
  diverges from the parent's history.
- Checkpoint pause window + fork-to-interactive + rollback-to-interactive measured and
  logged in every gate (CI-relative numbers for the ADR; the <100ms charter figure stays a
  bare-metal target).

## Out of scope (ADR-0006 records)

Cross-node fork; lineage-aware tiering/GC (fork parents are simply pinned HOT); fork of
WARM/COLD sandboxes (resume first); checkpoint pruning/retention policies; overlayfs/EROFS
rootfs sharing (04 #7); GPU state. M6 does not exist — after M5 the roadmap is debt
burn-down and bare-metal measurement.
