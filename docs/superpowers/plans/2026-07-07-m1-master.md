# EmberVM M1 (Single-Node MVP) — Master Spec

> **For agentic workers:** This is the master spec for milestone M1 (weeks 3-6 per docs/zh/03 §3).
> M1 spans six subsystems; per the writing-plans scope rule each phase gets its own detailed
> plan (`2026-07-07-m1-phaseN-*.md`) written just-in-time and executed with
> superpowers:subagent-driven-development or inline. This file locks the cross-phase design
> decisions and interfaces so phase plans cannot drift.

**Goal:** A single cloud VM runs the full EmberVM sandbox lifecycle over REST: build a template
from a Docker image, create/exec/pause/resume/snapshot/kill microVM sandboxes, 20 concurrent,
hot restore <1s including a 15GB data disk.

**Architecture:** API Server (Gin + PostgreSQL) drives a Node Agent (root daemon) through a Go
interface with two wirings (in-proc for `embervm dev`, HTTP-over-UDS for split mode). The Node
Agent owns Firecracker processes, jailer, netns pool, cgroup v2, the M0 uffd-handler, and a
storage backend (ZFS clone or plain-dir fallback). guestd runs as PID 1 inside every
template-built rootfs and serves exec/files/health on :7777.

**Tech Stack:** Go 1.24, Gin, pgx/v5, google/go-containerregistry (crane), golang.org/x/sys,
Firecracker v1.16.1 (N-1: v1.15.1), ZFS (OpenZFS 2.2), PostgreSQL 16.

## Global Constraints

- M1 exit criteria (docs/zh/03 §3, verbatim): 单机 20 并发; 热恢复 < 1s（含 15GB 数据盘）; 单台云 VM 可完整部署.
- GitHub Actions CI is the ONLY real test environment (macOS host has no KVM). Local = `make lint`, `GOOS=linux go build ./...`, `go test ./pkg/...`.
- Producers are the source of truth for JSON schemas; consumers mirror them exactly (M0 lesson).
- e2b-dev/infra: Apache-2.0 parts may be adapted with attribution in THIRD_PARTY_NOTICES.md; BSL parts are design-reference ONLY (docs/zh/05 §2).
- Data disk = raw file on a ZFS dataset, never zvol; `recordsize=16k..64k`, `primarycache=metadata` (docs/zh/04 §1 verdict).
- Firecracker `io_engine=Sync`; every snapshot manifest carries `(fc_version, host_kernel_version)` and `snapshot_format_version` from v0.1 (docs/zh/05 §4).
- v0.x REST API may break, but changes must be listed in CHANGELOG.md (docs/zh/05 §4).
- M0 bench suite (probe-server rootfs, restore-matrix, zfs-compare, genreport) must keep working unchanged — template rootfs is a separate artifact.
- unix socket paths in tests must stay under 104 bytes (macOS limit) — use short temp dirs.
- Boundaries: uffd chunk store/diff snapshots/WS trace → M2; archive tiers → M3; multi-node, Gateway, Redis, 50-concurrency → M4. `pkg/uffd` is consumed as-is from M0.

## Design decisions (resolving the six gaps the docs leave open)

**D1 — REST API v0** (docs list only the verbs create/pause/resume/snapshot/kill + auth/quota):

```
POST   /v0/templates                {name, image}                → 202 template (state=BUILDING)
GET    /v0/templates                                             → 200 [template]
GET    /v0/templates/{id}                                        → 200 template
DELETE /v0/templates/{id}                                        → 204
POST   /v0/sandboxes                {template_id, vcpus?, memory_mib?, data_disk_gib?} → 201 sandbox
GET    /v0/sandboxes[?state=]                                    → 200 [sandbox]
GET    /v0/sandboxes/{id}                                        → 200 sandbox
POST   /v0/sandboxes/{id}/pause                                  → 200 sandbox (PAUSED_HOT)
POST   /v0/sandboxes/{id}/resume                                 → 200 sandbox (RUNNING)
POST   /v0/sandboxes/{id}/snapshot                               → 200 {snapshot_id} (pause+snap+resume)
DELETE /v0/sandboxes/{id}                                        → 204 (kill)
POST   /v0/sandboxes/{id}/exec      guestapi.ExecRequest         → 200 guestapi.ExecResponse (proxy)
GET    /v0/sandboxes/{id}/files?path=/abs                        → 200 bytes (proxy)
PUT    /v0/sandboxes/{id}/files?path=/abs[&mode=0644]            → 204 (proxy)
GET    /healthz                                                  → 200 {ok:true}
```
Auth: `Authorization: Bearer <token>`; tokens + per-token `max_sandboxes` quota from config file.
Errors: `{"error": "..."}` with 4xx/5xx.

**D2 — apiserver↔nodeagent contract:** Go interface `pkg/nodeapi.Agent`
(CreateSandbox/StopSandbox/PauseSandbox/ResumeSandbox/SnapshotSandbox/Status/BuildTemplate),
wired in-proc for `embervm dev`; split mode = HTTP/JSON over unix socket
`/run/embervm/nodeagent.sock` mirroring the interface 1:1. gRPC deferred to M4 (ADR-0002).

**D3 — guestd v0:** static Go binary, PID 1 in template rootfs (`init=/usr/local/bin/guestd`).
PID 1 duties: mount proc/sysfs/devtmpfs/tmpfs, then fork self (`EMBERVM_GUESTD_CHILD=1`) and
reap orphans; the child serves HTTP on `0.0.0.0:7777`:
`GET /healthz` → `{ok, seq, pid, version}` (seq = per-process monotone counter — restore
continuity assert, same semantics as M0 probe-server); `POST /exec` (buffered, process-group
kill on timeout); `GET|PUT /files?path=`. Wire types in `pkg/guestapi` shared with the
apiserver proxy. Bench rootfs keeps probe-server; only template-built rootfs gets guestd.

**D4 — Template pipeline (Docker image → microVM rootfs):** pull+flatten via
go-containerregistry (`crane export`, daemonless) → staging dir → inject guestd → write
`/etc/resolv.conf`, image ENV/ENTRYPOINT/CMD/WORKDIR recorded into `/etc/embervm/image.json`
(guestd exec uses them as defaults) → `mkfs.ext4 -d` (root not required). squashfs+overlayfs
base-sharing (docs/zh/02 §2.3) is an M1-stretch/M2 optimization; ext4 satisfies the M1
charter ("Docker 镜像 → ext4/squashfs rootfs"). Guest kernel = M0 vmlinux-6.1.155; boot args per
docs/zh/04 §5: `reboot=k panic=1 8250.nr_uarts=0 swiotlb=noforce` + `ip=` static config.

**D5 — ZFS dataset layout** (pool name configurable, default `embervm`):
```
<pool>/templates/<template_id>        rootfs.ext4 + template.json     snapshot @final
<pool>/sandboxes/<sandbox_id>         = zfs clone <pool>/templates/<tid>@final
                                       + data.raw (sparse, data_disk_gib, default 15)
pause:   zfs snapshot <pool>/sandboxes/<id>@p<N>; memfile+snapfile+manifest.json in dataset dir
```
Properties: `recordsize=16k`, `primarycache=metadata`, `compression=lz4`, `sync=standard`.
Behind `pkg/storage.Backend` (ZFS impl + plain-directory impl for dev/tests; plain uses sparse
copies). CI creates a loop-backed pool exactly like test/bench/zfs-compare.sh `ensure_zfs`.

**D6 — PostgreSQL schema v0** (pgx/v5, embedded SQL migrations via embed.FS, no ORM):
```sql
templates(id uuid pk, name text unique, image text, state text, error text,
          created_at timestamptz, ready_at timestamptz)
sandboxes(id uuid pk, template_id uuid fk, state text, vcpus int, memory_mib int,
          data_disk_gib int, netns text, owner text, error text,
          created_at timestamptz, updated_at timestamptz, paused_at timestamptz)
sandbox_events(id bigserial pk, sandbox_id uuid, from_state text, to_state text,
               at timestamptz default now(), detail jsonb)
```
PostgreSQL is the single source of truth (docs/zh/04 §6: no homegrown distributed state).
Redis: not used in M1 (Gateway routing is M4) — recorded in ADR-0002.

**D7 — `embervm dev`:** `cmd/embervm` CLI, `dev` subcommand = migrations + apiserver +
in-proc nodeagent in one root process. Requires `--database-url` (default
`postgres:///embervm`); `deploy/singlenode/install.sh` provisions PostgreSQL via apt on a
stock Ubuntu 24.04 cloud VM (exit criterion 3). No embedded-postgres dependency.

**D8 — Lifecycle state machine (`pkg/lifecycle`):** M1 states
`PENDING → STARTING → RUNNING ⇄ (PAUSING → PAUSED_HOT → RESUMING) → STOPPING → STOPPED`,
plus `FAILED` from any active state. Transition table enforced in one place; every transition
appends to sandbox_events. WARM/COLD/RECYCLED enter in M3 (names reserved in the type).

**D9 — Hot resume path:** new FC process in the sandbox's existing netns, `PUT /snapshot/load`
with `mem_backend=Uffd` served by M0 `cmd/uffd-handler` (default mode uffd-prefetch, config
`restore_mode`), drives point at rootfs.ext4 + data.raw inside the dataset (O(1) attach —
data disk never enters the resume critical path, docs/zh/02 §1). Per-VM handler process +
watchdog restart per docs/zh/04 §6.

**D10 — Node Agent internals:** long-running root daemon; netns pool pre-created at startup
(config `netns_pool`, default 24 — covers the 20-concurrency criterion; naming `ember<N>`
continues M0's scheme, tap0 172.16.0.1/30, guest always 172.16.0.2 — identical-IP-after-clone
is BY DESIGN, isolation via per-sandbox netns+NAT, docs/zh/02 §4). Host reaches guestd by
dialing through the sandbox netns (custom dialer doing setns, or `ip netns exec` fallback —
Phase 4 plan decides after reading scripts/setup-network.sh). cgroup v2 slice per sandbox;
jailer on by default with per-VM uid/gid, `--new-pid-ns`, cgroup-version 2.

## Phases

Each phase: write phase plan → implement (TDD where the surface is testable without KVM) →
`make lint && make test && GOOS=linux go build ./...` locally → push → CI green before the
next phase. Exit-criteria measurements are CI-relative per ADR-0001 (bare metal still M1-later
task tracked separately).
