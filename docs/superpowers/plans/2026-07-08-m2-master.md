# EmberVM M2 (Second-Level Restore Pipeline) — Master Spec

> **For agentic workers:** This is the master spec for milestone M2 (weeks 7-10 per docs/zh/03 §3).
> M2 spans five subsystems; per the writing-plans scope rule each phase gets its own detailed
> plan (`2026-07-08-m2-phaseN-*.md`) written just-in-time and executed with
> superpowers:subagent-driven-development or inline. This file locks the cross-phase design
> decisions and interfaces so phase plans cannot drift.

**Goal:** Snapshots become chunked, compressed, content-addressed, layered (diff) objects that
write through to an S3-compatible L1 store on pause, and restore through a working-set-first
uffd pipeline: hot restore P50 < 500ms, warm restore (from L1) P99 < 3s, and a full
pause→upload→restore-on-another-node chain.

**Architecture:** A new `pkg/memsnap` defines the chunked memfile format (fixed 16KiB chunks,
lz4, zero-skip, per-chunk SHA-256) and layered manifests (full + diff). `pkg/chunkstore` stores
chunks content-addressed (local dir + S3/MinIO backends, parallel transfer). The uffd handler
gains a `chunked` mode that resolves faults chunk-by-chunk (local-first, L1 fallback), records a
working-set trace on first resume, and on later resumes prefetches the WS eagerly then backfills
the rest in the background (REAP + FaaSnap). The node agent boots VMs with dirty-page tracking,
turns every pause into a (Full first, Diff after) chunkified layer pushed to L1, ships data-disk
deltas as `zfs send -i` streams, and can rebuild a sandbox on a node that has never seen it.

**Tech Stack:** Go 1.25, github.com/pierrec/lz4/v4, github.com/minio/minio-go/v7 (S3),
crypto/sha256 (stdlib), Firecracker v1.16.1 (`track_dirty_pages`, `snapshot_type=Diff`),
OpenZFS 2.2 (`zfs send -i` / `zfs receive -o origin=`), MinIO (CI L1).

## Design decisions (ADR-0003 material)

**D1 — Chunk format (`pkg/memsnap`):** fixed-size **16 KiB** chunks (4 guest pages), NOT
content-defined — a page fault must map offset→chunk in O(1); CDC dedup belongs to the store
layer and its Xet-style evolution is M3. Per chunk: SHA-256 over the *uncompressed* bytes; all
zero chunks are recorded as `zero:true` and never stored (restored via UFFDIO_ZEROPAGE). Chunks
compress independently with lz4 block format; store the compressed form only when it is smaller
(`clen < ulen`), else raw (`codec:"raw"` vs `"lz4"`).

**D2 — Layer manifest (JSON, one per snapshot layer):**

```json
{
  "format_version": 1,
  "layer_id": "p1", "parent": "" | "p0",
  "kind": "full" | "diff",
  "fc_version": "v1.16.1", "kernel_version": "6.1.155",
  "mem_size_bytes": 2147483648, "chunk_size": 16384,
  "created_at": "...",
  "chunks": [ {"i": 0, "h": "sha256hex", "z": false, "c": "lz4", "ul": 16384, "cl": 5121} ]
}
```

A full layer lists every chunk index (zero chunks included, with `z:true, h:""`). A diff layer
lists only chunk indices it overwrites. `Resolve(layers []Manifest) []ChunkRef` walks newest→
oldest to produce the flat restore view. Trailing partial chunk (mem size not divisible by 16
KiB) is legal: `ul < chunk_size` only on the last chunk.

**D3 — Diff extraction:** boot AND every snapshot-load pass `track_dirty_pages:true`. First
pause takes `snapshot_type:"Full"`; later pauses take `"Diff"`. Firecracker writes a sparse diff
memfile (only dirty 4 KiB pages are data); the writer walks it with `SEEK_DATA`/`SEEK_HOLE`.
Because chunks are 4 pages, a partially-dirty chunk is **merged with the resolved parent chain's
content at write time** (dirty extents overlaid onto the parent chunk) so manifests always hold
self-contained chunks and restore stays newest-wins — recording the raw diff chunk would clobber
clean parent pages with hole-zeros (found by TestWriteDiffLayerPartialDirtyMerge). This requires
page-granular hole reporting: snapshot staging dirs must be ext4/tmpfs (agent probes at startup;
ZFS recordsize=16k rounds extents up and APFS reports materialized whole-file extents — diff
extraction is linux-only, `WriteDiffLayer` errors elsewhere and its tests skip off-linux).

**D4 — Chunk store (`pkg/chunkstore`):** content-addressed by SHA-256 hex.
`Store` interface: `Put(ctx, hash, r io.Reader, size) (written bool, err)` (no-op when present —
dedup is measured here), `Get(ctx, hash) (io.ReadCloser, error)`, `Has`, `Delete`. Backends:
`Dir` (local, sharded `objects/ab/abcdef...`, atomic rename) and `S3` (minio-go; bucket +
prefix; creds via env `EMBERVM_L1_{ENDPOINT,BUCKET,ACCESS_KEY,SECRET_KEY}`). `Copier` fans
out Put/Get with bounded parallelism (default 16) for write-through and backfill. SHA-256 now;
Blake3/Merkle is the M3 repo, per charter line "FastCDC → M3 evolution".

**D5 — uffd handler v2 (`pkg/uffd` + `cmd/uffd-handler`):** new `--mode chunked` with
`--manifest-dir <dir>` (contains `layer-*.json` resolved chain), `--store <dir>` (local chunk
dir), optional `--l1` (S3 env config) and `--ws <ws.json path>`. Fault → chunk index →
local store read → (miss) L1 GET write-through local → lz4 decode → `UFFDIO_COPY` of the whole
chunk; zero chunk → `UFFDIO_ZEROPAGE`. WS trace: first-touch chunk indices in fault order,
written to `--ws` path at handler exit (VM teardown) — `{"format_version":1,"chunks":[...]}`.
When the WS file exists at startup the handler eagerly loads WS chunks first (sequential,
batched), then backfills ALL remaining chunks in the background at `--prefetch-block`
granularity while serving faults with priority (FaaSnap concurrent paging; fixes the M0
cold-cache regression where whole-file prefetch contended with demand faults). Modes
`lazy|prefetch` stay byte-identical to M0. Watchdog: nodeagent already kills/restarts the
handler process per sandbox; v2 adds `--parent-pid` self-exit so an orphaned handler cannot
leak when the agent dies.

**D6 — Node agent pause path:** after `CreateSnapshot` → `memsnap.WriteLayer` (chunkify into
node-local store + manifest) → `chunkstore.Copier.Push` new chunks + manifest + snapfile + WS
file to L1 (write-through; pause is not "done" until L1 push succeeds — RPO per docs/zh/02 §3)
→ ZFS: `zfs snapshot data@p<N>` + `zfs send -i <prev> | store as object disk-p<N>.zstream`
(first pause sends `-i template@final`). Raw memfile/diff file is deleted after chunkify (the
chunk store is now the source of truth). Plain (non-ZFS) backend skips disk increments and
uploads nothing for the disk (dev-only path, documented).

**D7 — Node agent resume path:** ensure snapfile + manifests + WS local (pull from L1 if
missing) → start uffd-handler v2 (chunked) → `LoadSnapshot(mem_backend=Uffd,
track_dirty_pages=true)` → handler WS-prefetch foreground, background backfill → after guest
answers, agent POSTs `guestd /resumed`. Restore-on-node-B (`RestoreSandbox(id)` on an agent
that never ran it): pull manifests + snapfile + WS from L1, `zfs receive -o
origin=<pool>/templates/<tid>@final` the disk stream chain into a fresh dataset, allocate a
netns from the pool, then the normal resume path with an empty local chunk store (memory pulls
from L1 on WS-prefetch + faults). Template rootfs is node-global (M1 invariant): node B builds
or receives the template dataset out-of-band before restore; the e2e provisions it by building
the same template.

**D8 — guestd correctness hooks:** `POST /resumed` bumps a `resumes` counter (returned by
`/healthz`), best-effort re-arms the guest clock from the hypervisor (kvm-clock is already
authoritative; the hook exists for token refresh / user scripts: exec
`/etc/embervm/resume-hook` if present, 5s timeout, non-fatal). VMGenID: FC v1.16 injects it on
snapshot load and the 6.1 guest kernel reseeds its RNG — e2e asserts `/proc/sys/kernel/random/
uuid` differs across a resume while seq continuity holds (RNG reseed observable, no new code).

**D9 — What the apiserver sees:** no REST surface change in M2. `pause` now implies
write-through-L1 layered snapshot; `snapshot` verb unchanged (Full + resume). Cross-node
restore is exercised at the nodeagent layer by tests (scheduler owns it in M4). CHANGELOG
records the storage-format change (`snapshot_format_version: 1`  in manifests).

**D10 — e2e-m2 exit gates (`.github/workflows/e2e-m2.yml`):** MinIO server binary on the
runner (localhost:9000) as L1; loop ZFS pool as in e2e-m1.
- `TestHotRestoreP50` — 2 GiB VM, warm it, then ≥15 pause/resume cycles (chunked+WS);
  P50(load→interactive) < 500ms. Interactive = guestd healthz answers with seq+1.
- `TestWarmRestoreP99` — ≥10 cycles: pause → push L1 → wipe node-local chunk store → resume
  (WS + faults pull from L1); P99 < 3s.
- `TestCrossNodeRestore` — agent A: create, write marker, pause (write-through). Agent B
  (fresh WorkDir, fresh pool dataset tree, own netns): `RestoreSandbox` from L1 only; assert
  seq continuity, marker file, `resumes` counter == A's resumes+1.
- `TestDiffChain` — full → dirty ~64 MiB → diff → dirty → diff; restore newest; assert all
  markers; assert `diff layer stored bytes < 25% of full layer stored bytes`.
- Dedup report (non-gating): second sandbox from the same template, log Put-dedup ratio.
- All tests greppable `--- PASS:` in the workflow (M1 lesson), absolute asset paths.
