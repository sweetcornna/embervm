# EmberVM M3 (Tiered Archive & Lifecycle Engine) — Master Spec

> **For agentic workers:** This is the master spec for milestone M3 (weeks 11-13 per docs/zh/03 §3).
> Per the writing-plans scope rule each phase gets its own detailed plan written just-in-time and
> executed inline (superpowers:executing-plans) or via subagents. This file locks the cross-phase
> design decisions and interfaces so phase plans cannot drift.

**Goal:** Paused sandboxes flow automatically through storage tiers —
PAUSED_HOT → PAUSED_WARM → ARCHIVED_COLD → RECYCLED — on configurable TTLs, with cold-archive
restore < 10s to interactive, wake-histogram pre-warm prediction, selective (artifacts-only)
restore from RECYCLED, and per-sandbox cost reporting by tier.

**Architecture:** A lifecycle engine goroutine in the control plane scans PostgreSQL on a tick
and drives transitions: WARM releases all node-local resources (the L1 write-through from M2
already holds everything); COLD compacts the memory layer chain into a **synthetic full
manifest (metadata-only — chunks are content-addressed, so a merged layer is just a new
manifest)** and copies referenced objects to a second, cheaper object store; RECYCLED extracts
configured artifact paths from the disk (host-side read-only loop mount) into a tarball and
deletes the snapshot, with a mark-and-sweep chunk GC reclaiming unreferenced chunks. Resume
works from every tier through the M2 restore path, pointed at the right store; a pre-warm
predictor (Azure "Serverless in the Wild" histogram policy) pulls the working set back to the
node before the predicted wake.

**Tech Stack:** Go 1.25, existing pkg/{memsnap,chunkstore,uffd,nodeagent,controlplane,
lifecycle,nodeapi}, PostgreSQL 16 (pgx/v5), MinIO ×2 buckets in CI (warm L1 + cold L2),
OpenZFS 2.2, Firecracker v1.16.1.

## Global Constraints

- M3 charter scope (docs/zh/03 §3, verbatim): 生命周期引擎 HOT→WARM→COLD→RECYCLED 定时流转,
  TTL 可配; 冷层转储 + 回填热点 chunk 优先; 唤醒直方图 pre-warm 调度(《04》#5): 预测唤醒前拉回
  本地 + 预解压 WS; 选择性恢复(Manus 式): RECYCLED 后仅回灌 artifacts/指定目录; 成本报表:
  每沙箱存储占用与层级分布.
- M3 exit criteria (verbatim): 冷归档恢复 < 10s 可交互; 归档成本达标.
- Restore SLOs stay intact per tier (docs/zh/02 §3): HOT P50<500ms, WARM P99<3s (both proven
  in M2 and must not regress), COLD <10s.
- GitHub Actions CI is the ONLY real test environment; latency gates CI-relative (ADR-0001).
- Every exit-criteria test greppable `--- PASS:`; absolute asset paths (M1 lesson).
- Seq continuity assertions are monotone-above-snapshot, never exact +1 (M2 lesson).
- M0 bench, M1 e2e, M2 e2e must stay green on the same commit.
- Carried M2 debts to land here: synthetic-full compaction (memory chain), chunk GC. Rolling
  WS refresh and Xet CDC/Merkle/block-aggregation remain deferred (recorded in ADR-0004);
  ARIMA fallback for sparse wake data is deferred — fixed keep-alive is the fallback.
- MinIO is CI-only tooling; production L1/L2 guidance stays Garage/SeaweedFS/Hetzner OS/B2
  (docs/zh/04 §1: minio/minio archived 2026-04).

## Design decisions (ADR-0004 material)

**D1 — Tier semantics (what each state means for bytes):**

| State | node-local (L0) | warm store (L1) | cold store (L2) |
|---|---|---|---|
| PAUSED_HOT | dataset + workdir + chunk cache | everything (M2 write-through) | — |
| PAUSED_WARM | nothing | everything | — |
| ARCHIVED_COLD | nothing | nothing (moved) | synthetic full + chain objects |
| RECYCLED | nothing | nothing | artifacts.tar.zst only |

Transitions: `PAUSED_HOT --ttl_warm--> PAUSED_WARM --ttl_cold--> ARCHIVED_COLD
--ttl_recycle--> RECYCLED`. Resume is legal from HOT/WARM/COLD (returns to RUNNING through
RESUMING); RECYCLED is terminal for the sandbox — its artifacts seed a NEW sandbox
(selective restore). lifecycle.transitions gains: PAUSED_HOT→PAUSED_WARM,
PAUSED_WARM→{RESUMING, ARCHIVED_COLD, STOPPING}, ARCHIVED_COLD→{RESUMING, RECYCLED, STOPPING},
RECYCLED→STOPPING. PAUSED_WARM/ARCHIVED_COLD join activeStates (may FAIL).

**D2 — Lifecycle engine (`pkg/controlplane/engine.go`):** a goroutine owned by the apiserver
(and `embervm dev`), tick default 30s (configurable), TTLs from config
`{ttl_warm, ttl_cold, ttl_recycle}` (0 disables a transition). Scan = one SQL query per tick
over paused states + paused_at. Each transition is executed then recorded via the existing
SetSandboxState/sandbox_events plumbing; failures mark FAILED with the error (and are retried
never — operator visible). The engine serializes per sandbox; a resume request racing a
transition is resolved by the state machine (SetSandboxState validates from-state
compare-and-swap, which the store already enforces via WHERE state=$from).

**D3 — WARM = release node-local resources.** New nodeapi verb
`ReleaseLocal(ctx, sandboxID) error`: kills nothing (sandbox is paused, FC already dead),
verifies the L1 descriptor exists (refuses otherwise — WARM without L1 is data loss), then
destroys the dataset (`DestroySandbox`), removes the workdir, releases the netns lease, and
drops the in-memory sandbox entry. The node-local chunk cache is NOT purged per-sandbox
(content-addressed chunks are shared; L0 eviction is an LRU concern deferred to M4 and noted
in ADR-0004). Resume from WARM = M2 `RestoreSandbox` (already node-agnostic), now exposed as
nodeapi verb `RestoreSandbox(ctx, sandboxID, tier string) (SandboxStatus, error)`.

**D4 — COLD = synthetic full + move to the cold store.** Control-plane-only operation (pure
store-to-store, no node involved):
1. `memsnap.Synthesize(view, layerID)` — new function: flatten the resolved chain into ONE
   full-kind manifest (`layer-cold.json`). Metadata-only; zero chunk movement. This lands the
   M2 debt "diff chains grow unboundedly": cold restores read exactly one layer.
2. Copy referenced chunks L1→L2 (`chunkstore.Copier`, dedup-aware), plus snapfile of the
   newest layer, ws.json, the disk delta chain (zstream objects, unchanged — zfs receive
   replays it; disk synthetic full needs a live dataset and is deferred with rationale), and a
   rewritten `snapshot.json` (layers: ["cold"], tier: "cold", disk_layers preserved).
3. Delete the sandbox's objects from L1 (chunks are NOT deleted per-sandbox — the GC owns
   chunk deletion; L1 manifests/snapfiles/descriptor are removed).
Cold store config: `EMBERVM_COLD_*` env, same schema as `EMBERVM_L1_*` (chunkstore.ColdFromEnv
alongside L1FromEnv; both may point at the same MinIO with different buckets in CI).

**D5 — Chunk GC (`pkg/chunkstore/gc.go`):** mark-and-sweep over one Backend: List all
`meta/sandboxes/*/layer-*.json` + `meta/templates/*` manifests → live hash set → delete
unreferenced chunks. Requires a new `List(ctx, prefix string) ([]string, error)` on Backend
(Dir: filepath.WalkDir; S3: ListObjects). Runs after RECYCLE transitions (engine-triggered)
and is invocable standalone. Safety: GC lists manifests BEFORE chunks and never deletes a
chunk younger than a grace window (mtime < now-1h default, configurable to 0 in tests) so an
in-flight pause's chunks (uploaded before their manifest) survive.

**D6 — RECYCLED = artifacts out, everything else deleted.** Sandboxes get an
`artifact_paths []string` column (set at create, default empty = keep nothing). New nodeapi
verb `ExtractArtifacts(ctx, sandboxID, paths []string) error` (needs a node: ZFS + loop
mounts): receive the disk chain from cold into a scratch dataset (M2 Replicator), loop-mount
rootfs.ext4 and data.raw read-only on the host, tar the requested paths (missing paths are
skipped, not errors), `PutObject cold: sandboxes/<id>/artifacts.tar.zst` (zstd via
klauspost/compress — pure Go, no cgo), destroy the scratch dataset. Then the engine deletes
the sandbox's cold snapshot objects and triggers GC. Selective restore: REST
`POST /v0/sandboxes/{id}/restore-artifacts {new_sandbox_id?}` creates a fresh sandbox from the
same template and untars artifacts into it via guestd (host pulls the tarball, streams it with
the existing WriteFile + an exec `tar -xzf`). RECYCLED sandbox rows are kept (state RECYCLED)
for audit; the new sandbox is a new row.

**D7 — Resume-from-tier plumbing.** The descriptor (`snapshot.json`) gains
`"tier": "hot"|"warm"|"cold"` and `"disk_layers": [...]` (explicit, no longer implied by
layers). nodeagent `RestoreSandbox` picks the source Backend by tier and sets the uffd
handler's child env (`EMBERVM_L1_*` values pointing at the chosen store) so faults and WS
prefetch pull from the right place — hot-chunk-first is inherited from M2's WS-first pipeline
(the charter's 回填热点 chunk 优先). After a cold resume the next pause writes through to L1
again (tier resets to hot chain semantics with a fresh Full layer, because the synthetic-full
parent lives in L2; simplest correct rule: resume-from-cold → next pause takes Full).

**D8 — Pre-warm predictor (`pkg/prewarm`):** per-sandbox wake-interval histogram fed from
sandbox_events (PAUSED_*→RESUMING edges). Policy (Serverless in the Wild): with ≥3 samples
and CV ≤ 2, predict the next wake at `paused_at + P5(intervals)`; the engine pre-warms a
WARM/COLD sandbox when `now ≥ predicted - prewarm_lead` (default lead 60s): pull WS-listed
chunks + manifests + snapfile to the node-local store (Copier L2/L1→L0) and mark the sandbox
`prewarmed_at`. With <3 samples or CV > 2: no prediction (fixed keep-alive = the TTLs
themselves; ARIMA deferred). Pre-decompression is deliberately skipped: lz4 decode ~5 GB/s/core
is not the bottleneck (ADR-0004 records this). Histogram math is pure and unit-tested;
the engine integration is exercised in e2e (report: prewarmed resume vs cold resume).

**D9 — Cost report.** `GET /v0/sandboxes/{id}/storage` →
`{tier, logical_bytes, stored_bytes_by_tier: {l1, cold}, chunk_count, dedup_ratio}` computed
on demand from manifests (logical = Σ ULen of newest-wins view; stored = Σ CLen of referenced
chunks actually present per store; artifacts size for RECYCLED). `GET /v0/storage-report` →
per-sandbox rows + totals. Computed, not cached (CI scale; caching is an optimization we
don't need yet).

**D10 — e2e-m3 exit gates (`.github/workflows/e2e-m3.yml`):** loop ZFS pool + one MinIO with
two buckets (embervm-l1, embervm-cold).
- `TestLifecycleTTLFlow` — tiny TTLs (seconds); create → pause → engine drives
  HOT→WARM→COLD automatically (poll states via store); resume from WARM (works, < 3s);
  re-pause; drive to COLD; **resume from COLD < 10s interactive** (THE exit gate) with seq
  continuity + markers; re-pause → COLD → RECYCLE with artifact_paths set → artifacts.tar
  exists in cold store, snapshot objects gone, GC ran (unreferenced chunk count reported);
  selective restore into a new sandbox → artifact file readable in the new guest.
- `TestSyntheticFullManifest` (unit-level but listed as a gate): chain of 3 layers →
  Synthesize → one full manifest, byte-identical reconstruction, and cold restore reads
  exactly 1 memory manifest.
- `TestArchiveCostReport` — the 归档成本达标 proxy, CI-relative: cold stored_bytes ≤ 60% of
  logical bytes for the test workload (zero-skip + lz4 + dedup + synthetic-full dropping
  overwritten chunks), storage report endpoint returns consistent numbers; log everything.
- `TestPrewarmPolicy` (unit, in gate list) + prewarmed-resume-vs-cold report in
  TestLifecycleTTLFlow (non-gating latency comparison, logged).
- All gates grepped `--- PASS:`.
