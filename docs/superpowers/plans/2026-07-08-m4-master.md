# EmberVM M4 (Elasticity & Production Hardening) — Master Spec

> **For agentic workers:** Master spec for milestone M4 (weeks 14-16 per docs/zh/03 §3). Phases
> execute inline (superpowers:executing-plans); this file locks cross-phase decisions so phase
> work cannot drift. Predecessors: ADR-0002 (M1), ADR-0003 (M2), ADR-0004 (M3).

**Goal:** A 3-node cluster where killing any worker leaves every sandbox recoverable on another
node, with jailer-hardened Firecracker, sub-500ms sandbox creation from template snapshots,
50-per-node concurrency, a gateway proxy, Prometheus observability — G1–G6 accepted, tagged as
the open-source v0.1 release.

**Architecture:** The control plane grows a scheduler: a `nodes` registry (PostgreSQL, polled
heartbeats, eviction on missed polls) and placement (sticky-to-previous-node, else bin-packing
by free memory); sandboxes carry `node_id` and every agent call routes through a node registry
of `nodeapi` clients. Firecracker moves under **jailer** (chroot + per-VM uid/gid + netns +
default seccomp) with the sandbox dataset and snap dir bind-mounted into the chroot — snapshot
drive paths become chroot-relative (`/data/...`), which retires M3's mountpoint pinning AND
unlocks fast-create: at template build a golden sandbox is booted once and paused, and
`CreateSandbox` restores that template snapshot (~150ms hot path) instead of cold-booting.
Balloon devices enable memory reclaim; an agent watchdog reaps dead FC/uffd processes and
flags the sandbox for restore (G5). The gateway is two chained reverse proxies (apiserver →
node agent → guest via the netns dialer), WebSocket-transparent. Prometheus metrics on both
planes.

**Tech Stack:** Go 1.25, existing pkg tree, Firecracker v1.16.1 jailer, PostgreSQL 16,
prometheus/client_golang, MinIO ×2 + loop ZFS + PG in CI.

## Design decisions (ADR-0005 material)

**D1 — Node registry & heartbeats (poll, not push).** `nodes` table:
`(id text pk, addr text, state text 'up'|'down', capacity_mib int, last_seen timestamptz)`.
Static membership from config (`--nodes id=addr,...` / env `EMBERVM_NODES`); `embervm dev`
auto-registers its in-proc agent as node "local". The scheduler goroutine polls every node's
`Healthz` (new lightweight nodeapi verb returning capacity + used MiB + sandbox count) each
tick (default 5s); `miss_threshold` (default 3) consecutive failures → state='down' + its
RUNNING sandboxes marked FAILED (their last write-through snapshot stays restorable), paused
ones untouched (they live in L1/L2 anyway). Recovery is restore-on-demand: resume of a sandbox
whose node is down re-places it on a healthy node — no mass migration storm.

**D2 — Placement: sticky, then bin-pack.** `sandboxes.node_id text` column. On create/resume:
prefer the previous node (L0 cache locality, docs/zh/02 §3 placement 粘性) when it is 'up' and
has free memory; else the up node with most free memory (bin-packing by the dominant
constraint, docs/zh/02 §2.1); none → 503. Free = capacity − Σ memory_mib of RUNNING/RESUMING
sandboxes on it (PG query; the in-memory view is a cache, PostgreSQL is the only truth —
Corrosion lesson, docs/zh/04 §6). apiserver keeps `map[nodeID]nodeapi.Agent` built from config.

**D3 — Jailer hardening (the M1-deferred debt, docs/zh/04 §5).** Every FC process runs under
`jailer --id <sandboxID> --uid <base+N> --gid <base+N> --chroot-base-dir <jail-root> --netns
/var/run/netns/<lease>`. Chroot layout (all bind mounts, torn down on stop/release):

```
<jail-root>/firecracker/<id>/root/
  data/  ← bind: sandbox dataset dir   (rootfs.ext4, data.raw)
  snap/  ← bind: workdir snap dir      (snapfile-*, uffd.sock, memfile-* staging)
```

FC sees drives as `/data/rootfs.ext4`, snapshots as `/snap/snapfile-pN`, the uffd socket as
`/snap/uffd.sock` (same inode as the host-side handler's listen path). Consequences: snapfile
drive paths are chroot-relative and IDENTICAL on every node and every sandbox — M3's
`SetSandboxMountpoint` pinning becomes unnecessary in jailed mode (kept for the unjailed dev
path) and fast-create (D4) becomes possible at all. Dataset files are chowned to the jail uid
at clone/restore. Default seccomp stays ON (no --no-seccomp); cgroup placement keeps our
existing controller (jailer cgroup flags unused). `Config.JailerBin` + `Jail bool` (default on
when the binary is present; e2e exercises both).

**D4 — Fast-create from template snapshots (G4 创建<500ms).** At template build: boot a golden
sandbox from the template once, wait for guestd, pause it through the normal chunked pipeline,
and record its artifacts under `templates/<tid>/snap/` in L1 (chunks live in the shared pool;
golden dataset snapshot `@golden` retained on the template dataset). CreateSandbox fast path
(when the template has a golden snapshot AND the requested geometry matches the golden one):
`zfs clone <template>@golden` + restore memory via the normal chunked resume against the
template's snapshot objects + fresh netns + `POST /resumed`. Cold boot remains for mismatched
geometry or missing golden snapshot. The guest is IDENTICAL across fast-created sandboxes by
design (same IP per netns isolation; VMGenID reseeds the RNG on load; guestd seq starts at the
golden value — continuity assertions remain monotone).

**D5 — Balloon + oversell (G4 50 并发).** Boot config gains a virtio-balloon device
(`amount_mib: 0`, `deflate_on_oom: true`); new nodeapi verb `SetBalloon(ctx, id, targetMiB)`
PATCHes it so operators (and the M5 density work) can reclaim idle guest memory. CPU oversell
is inherent (vCPUs are threads, not pinned); memory oversell rides zero-page lazy faulting.
Gate: `TestConcurrency50` — 50 × 256 MiB sandboxes on one agent, all exec-verified (logical
12.8 GiB on a 16 GiB runner; actual RSS far lower via lazy memory).

**D6 — Zombie reaping (G5).** The agent gains a watchdog goroutine (tick 5s): a sandbox whose
FC process has exited without a Stop/Pause → kill its uffd handler, release the lease, mark
the in-memory machine FAILED; `Status` surfaces it and the control plane's next poll writes it
through to PG. uffd handler death with a live VM (faults would hang forever) → kill the FC
process too, same path. Recovery = the sandbox's last snapshot restores on demand (D1). The
uffd side already self-exits via --parent-pid; this is the FC-side mirror.

**D7 — Gateway: two chained reverse proxies, WebSocket-transparent.**
`ANY /v0/sandboxes/{id}/proxy/{port}/*path` (apiserver, authenticated, owner-scoped) →
node agent `ANY /sandboxes/{id}/guest/{port}/*path` (nodeapi transport) → guest
`http://172.16.0.2:{port}` via the existing netns dialer. Both hops are
`httputil.ReverseProxy` (WS/Upgrade passes through since go1.12). Port allowlist per sandbox
deferred; the proxy only reaches the sandbox's own netns. Redis routing tables are not needed:
the sandbox row IS the routing table (node_id → registry client).

**D8 — Egress policy at the netns level.** CreateSandbox gains `egress: "nat"|"none"`
(default nat = today's MASQUERADE). "none" configures the lease without the NAT rule —
sandbox can talk to the host-side proxy targets only. The full zero-trust L7 egress proxy
(domain allowlists, credential injection, secret redaction — docs/zh/02 §4) is explicitly
deferred past v0.1 (ADR-0005): it is a product subsystem, not a hardening flag.

**D9 — Observability: Prometheus now, tracing when it pays.** `/metrics` (promhttp) on
apiserver and nodeagent daemon. Counters/histograms: restore duration by tier, create duration
by path (cold/fast), lifecycle transitions by edge, chunk store ops (puts/gets/dedup hits/GC
sweeps), proxy requests, watchdog reaps, engine tick errors. `deploy/grafana/embervm.json`
ships a starter dashboard. OTel tracing deferred (single-hop system; ADR-0005).

**D10 — e2e-m4 exit gates + v0.1 release.** Workflow boots PG + MinIO×2 + loop ZFS + THREE
nodeagent daemons (UDS; ZFS subtrees n1/n2/n3; netns ranges via pool offsets) + apiserver.
- `TestClusterKillNode` (THE exit gate): create sandboxes across nodes (placement observed),
  write markers, pause some / keep some running; `kill -9` one agent; scheduler marks it down;
  resume a paused sandbox that lived there → restored on a healthy node with continuity;
  running-on-dead-node sandbox → FAILED then resumed elsewhere from its last snapshot.
- `TestFastCreateUnder500ms` — golden-snapshot create P50 < 500ms to interactive (G4).
- `TestConcurrency50` — 50×256MiB on one node, exec-verified (G4).
- `TestJailedLifecycleKVM` — full chunked pause/resume/restore cycle UNDER jailer (chroot
  paths in snapfiles), plus the M1 raw path still green unjailed.
- `TestWatchdogReapsZombies` — kill FC behind the agent's back; sandbox turns FAILED; resume
  restores it (G5).
- Gateway proxy exercised in-flow (HTTP echo through both hops).
- G1-G6 acceptance recorded in `docs/acceptance-v0.1.md` mapping each goal to its CI evidence.
- Tag `v0.4.0-m4` + GitHub Release "EmberVM v0.1" on it (开源 v0.1 发布).
