# 沙箱隔离方案深度对比:Docker vs Firecracker 及 2026 全景

> 日期:2026-07-10 · 结论服务于 M6(运行时弹性资源分配)的底座决策
> 本文是 docs/zh/01 §5"竞品格局"的深化与 2026 年中更新,聚焦一个此前未单列的维度:**运行时弹性资源分配能力**。

## 0. 一句话结论

**Docker 打包的是"进程",Firecracker 打包的是"机器"。** 对"跑不可信 AI 代码 + 状态即资产(暂停/恢复/分支/迁移)"的定位,Firecracker 仍是正确底座;Docker 唯一的结构性优势——运行时资源伸缩——自 FC v1.14 引入 virtio-mem 后已可在 microVM 上补齐(本仓库钉的 v1.15.1/v1.16.1 均已支持,guest 内核 6.1.155 已编入驱动,均经二进制验证),而 FC 的两大优势(硬隔离、原生快照/diff/uffd)Docker 至今补不上(CRIU 六年未走出 experimental)。

## 1. Docker vs Firecracker 逐维度对比

| 维度 | Docker/runc | Firecracker | 判定 |
|---|---|---|---|
| 隔离边界 | namespace+cgroup+seccomp,共享宿主内核 | KVM 硬件虚拟化,每沙箱独立内核 | FC 胜 |
| 逃逸风险 | 内核 syscall 全暴露(~350 个),容器逃逸 CVE 常态化 | 攻击面 = VMM(约 5 万行 Rust)+ KVM ioctl | FC 胜 |
| 启动延迟 | 27–90ms | 冷启动 125–150ms;快照恢复 <500ms(本项目 CI 实测) | Docker 略胜,快照拉平 |
| 单机密度 | 开销≈0,千级容器 | VMM ~5MiB + guest 内核数十 MiB,50–150 VM/节点 | Docker 胜 |
| **内存快照/恢复** | CRIU:2026 年仍 experimental、单机、TCP/tty/seccomp 全是坑 | 原生 full/diff snapshot + uffd 自定义内存后端 | **FC 碾压** |
| **运行时改内存** | `docker update --memory` 即时生效 | balloon(只能收)+ virtio-mem 热插拔(v1.14+,双向,缩容立即归还宿主 + mprotect 硬保护) | 接近打平 |
| **运行时改 CPU** | `docker update --cpus` 即时生效 | vCPU 热插拔上游明确不做(#2609);等效方案 = 按上限开机 + cgroup `cpu.max` 配额 | Docker 胜,FC 可 95% 等效 |
| 暂停时的资源占用 | `docker pause` 后内存仍占着 | pause→快照落盘,算力占用**归零**,资源回池 | FC 胜(这也是弹性) |
| GPU | 原生 | PCIe/VFIO 直通 developer preview(v1.16 起推进中) | Docker 胜 |
| 生态 | 镜像/网络/编排/监控全家桶 | 自建(本项目 M1–M4 的工作) | Docker 胜 |
| 宿主要求 | 任何 Linux | 必须 KVM(裸金属/嵌套虚拟化) | Docker 胜 |

**关键洞察:两者不是互斥选择。** 行业统一模式(E2B、本项目均如此)是 **Docker 镜像作为模板分发格式 + Firecracker 作为运行时**——生态优势与隔离/快照优势同时拿到。真正的问题只是运行时层用哪个。

## 3. 商业平台格局(2026 年中,更新自 docs/zh/01 §5)

| 平台 | 隔离技术 | 特点 | 弹性/持久化语义 |
|---|---|---|---|
| E2B | Firecracker | SDK 生态最好,安全隔离卖点 | pause/resume,~150ms 创建 |
| Modal | gVisor + 自研内存快照 | **GPU 场景赢家**(T4→B200 全谱),快照带后台惰性恢复 | 容器级弹性 |
| Daytona | Docker(可选 Kata) | 27–90ms 最快创建;**快照不含内存态**,"恢复"是重启进程 | 容器级弹性,快照语义弱 |
| Morph Cloud | 完整 VM | Infinibranch 活体分支 <250ms,弹性最强 | 全虚拟化,密度/成本代价 |
| Fly Machines | Firecracker | suspend/resume 数百 ms,热/冷分层计费 | 机器语义 |
| Cloudflare Sandboxes | 容器 + Durable Object | idle 即眠,全盘 snapshot+fork | 恢复 ~2s |
| CodeSandbox | Firecracker | resume 均值 400ms/P99 2s,uffd fork <1.5s(本项目主要对标) | 机器语义 |
| Northflank | Kata + gVisor | BYOC,月 200 万工作负载,无限期持久环境 | 混合 |

格局对应:**要最快启动选容器系,要 GPU 选 gVisor 系,要状态即资产选 FC 系**——三条路线泾渭分明,没有平台同时占满三格。

## 4. 两个挑战者的深度分析

### 4.1 Cloud Hypervisor:弹性最强的 microVM,为什么不换

CH 赢的只有 vCPU 热插拔(guest 可见核数可变)。换底座的真实代价:

1. **无 uffd 内存后端(决定性)**——FC 恢复快照可指定 `MemBackend: Uffd`,由外部 handler 从内容寻址 chunk store 按需喂页;CH 恢复只能 mmap 内存文件,无外部 handler 接口。本项目 M2(chunked 惰性恢复+REAP 预取)、跨节点 L1 恢复、M5 fork 共享 chunk、零页跳过全建在这个接口上,换 CH 等于推倒重来。
2. **无 diff 快照链**——Full→Diff 分层写穿依赖 FC 的 `track_dirty_pages`,换 CH 后每次 pause 全量,写穿经济性崩塌(ADR-0003 作废)。
3. **无 jailer**——ADR-0005 的 chroot 相对路径设计(快照位置无关、golden fast-create、共享宿主 CI)需重造。
4. 波及面:pkg/uffd(12 文件)、pkg/memsnap、pkg/fcclient、nodeagent 全部恢复路径、M2-M5 四个已验收门禁。保守估计**数月重写**,换来的 vCPU 热插拔在 FC 上用 cgroup 配额几天可 95% 等效。

旁证:"状态即资产"赛道(E2B/CodeSandbox/Cursor/Fly/Lambda/Vercel Sandbox)**全部在 FC 上,无一家在 CH 上**——CH 的目标场景是长寿命通用云 VM,FC 的目标场景是海量短周期、快照为中心的负载。

### 4.2 gVisor:Modal 路线,为什么只作参照

gVisor 2026 年的 checkpoint/restore 相当成熟:自带后台异步恢复(缺页优先加载,等效 uffd 惰性恢复)、GPU 顺畅(nvproxy)、弹性=容器级。但:隔离低一档(用户态内核仍共享宿主内核真实攻击面)、syscall 密集负载 2–10x 开销、快照编排需自研(Modal 的护城河未开源)。对已建成 FC 栈的本项目是竞品参照而非迁移目标;**若未来 GPU 沙箱进入路线图,再启动 gVisor 双轨调研**。

## 5. 对 M6 的决策含义

1. 维持 Firecracker;运行时弹性走 **virtio-mem(内存)+ cgroup cpu.max(CPU)** 路线,详见 ADR-0007(待写)。
2. 版本条件已验证:FC v1.15.1/v1.16.1(virtio-mem 于 1.14 引入、1.15 修 KVM slot bug)、guest vmlinux-6.1.155 已编入 virtio_mem 驱动与 CPU 热插拔支持。
3. 唯一永久妥协:guest 可见 vCPU 数不可变(按上限开机,配额定义有效算力;guestd 内 CPU online/offline 为可选优化)。
4. 主要风险:上游文档未明说快照恢复后能否继续 `PATCH /hotplug/memory` —— M6 Phase 0 实验先行验证;失败则回退 balloon-headroom 方案(API 面不变)。

### 何时应重议本结论(反方条件)

- "guest 可见核数必须真实变化"成为硬需求(如按 `nproc` 授权的软件)→ CH 胜;
- GPU 沙箱进入近期路线图 → gVisor/Kata 路线更快;
- virtio-mem 与 uffd 恢复存在无法绕过的缺陷(Phase 0 验证)→ 回退 balloon-headroom;
- 若为零代码新项目(无 M2-M5 存量)→ CH 是认真候选。
