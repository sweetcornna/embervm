## 1. 项目背景与目标

AI Agent（Manus/Cursor 类）需要给每个会话一台完整、隔离、可休眠唤醒的云端计算机。商业方案（E2B/Modal/Daytona）存在成本、数据主权、大容量持久盘（>5GB）支持不足的问题；E2B 开源版自托管深度耦合 GCP+Nomad 且官方承认网络隔离/LB/部署未解决。**EmberVM 以 GitHub 开源项目（AGPL-3.0）形式自建一套为自托管而生的沙箱云软件服务**，任何人可在自己的裸金属/云服务器上部署与实测。

**核心目标（验收口径）：**

| # | 目标 | 指标 |
|---|---|---|
| G1 | 秒级恢复 | 热恢复 P50 < 500ms（P99 < 2s）；温恢复（含 10-20GB 数据盘）P99 < 3s 可交互 |
| G2 | 持久化存储 | 每沙箱 10-20GB 数据盘；pause 后数据零丢失（写穿 L1，RPO ≤ 5min）；跨节点可恢复 |
| G3 | 分层冷归档 | 热/温/冷/回收四级自动流转；冷层成本 ≤ $5/TB/月 |
| G4 | 弹性计算 | 沙箱创建 < 500ms；单节点 ≥ 50 并发；节点池水平扩容 |
| G5 | 稳定性 | 控制面可用性 ≥ 99.9%；uffd/进程僵死自动回收 |
| G6 | 开源可部署性 | 单台嵌套虚拟化云 VM 可跑 `embervm dev` 完整体验；GitHub Actions 免费 CI 跑真 microVM 测试 |

**非目标（本期不做）**：GPU 沙箱（实验 flag 卡位）、跨区域多活、对外商业化计费系统。

## 2. 技术路线（结论先行）

单机 ZFS 为中心 + 对象存储归档（CodeSandbox/E2B 混合式），全部选型经实测数据验证（《04》§1）：

- 运行时：Firecracker microVM（KVM），深度复用 e2b-dev/infra（Apache 部分可并入 AGPL 项目；BSL 部分只读不抄；其未解决的网络隔离/LB/部署三件事自研补齐）
- 热层：本地 NVMe ZFS——dataset + raw file（非 zvol）、clone O(1) 建盘、`zfs send -i` 增量
- 内存恢复：uffd 懒加载 + **WS 工作集预取（必选）** + 8-16KiB lz4 chunk + 零页跳过 + balloon
- 温/冷层：Garage/SeaweedFS 或 Hetzner Object Storage chunk 仓库（FastCDC 去重）→ lifecycle 转低价冷层
- 控制面：Go + PostgreSQL（托管或主从+WAL-G）+ Redis；调度粘性回原节点；不自研分布式状态同步

理由：延迟最优、组件最少、成本最低；CodeSandbox 已在 4-12GiB VM 生产验证 resume avg 400ms / P99 2s；10-20GB 数据盘因块设备天然 lazy 不进恢复关键路径。规模化后按需演进 JuiceFS 混合（跨机）或 Ceph（需专职存储运维）。

## 3. 里程碑计划

> 工期校准：16 周达成**内部 MVP**；从 MVP 到**可收费 beta / 生产级开源发布**合理预期 4-6 个月——demo 到生产的 70% 工作量在网络隔离、调度与可靠性加固（清单见《04》§5/§6）。

### M0：环境与原型验证（第 1-2 周）
- 双环境：1 台裸金属（Hetzner Auction €25-40/月，基准级数据）+ 1 台嵌套虚拟化云 VM（DigitalOcean/GCP Spot，验证"普通云服务器可跑"——开源核心承诺，矩阵见《06》）
- 跑通 Firecracker：内核 + rootfs 引导、netns/NAT、jailer
- **关键验证**：snapshot → uffd resume 延迟基线；ZFS raw file（对照复测 zvol）挂 virtio-blk；补齐《04》§9 数据缺口
- 产出：POC 脚本 + 基线数据报告（发布为公开 benchmark）

### M1：单机 MVP（第 3-6 周）
- Node Agent v0：创建/停止/暂停/恢复，cgroup v2、netns 管理
- 模板系统：Docker 镜像 → ext4/squashfs rootfs（复用 e2b template builder 思路）
- guestd v0：exec、文件读写、健康检查
- ZFS 集成：base clone、pause snapshot、resume clone
- API Server v0 + PostgreSQL：REST 全生命周期；`embervm dev` 单进程模式
- **退出标准**：单机 20 并发；热恢复 < 1s（含 15GB 数据盘）；单台云 VM 可完整部署

### M2：秒级恢复与快照管道（第 7-10 周）
- uffd handler（每 VM 独立进程）：8-16KiB lz4 chunk memfile、零页跳过、缺页回填、**WS trace 记录 + 顺序预取（必选）**、watchdog
- diff 快照：dirty-page bitmap 只存脏页；`zfs send -i` 数据盘增量；快照带 `(fc_ver, kernel_ver)` 标签
- chunk 仓库：FastCDC + 内容寻址 + Garage/SeaweedFS（或 Hetzner OS）；并行上传下载；pause 写穿 L1
- 正确性：VMGenID、chrony 校时、resume token 刷新 hook
- **退出标准**：pause→上传→异机 resume 全链路；热恢复 P50 < 500ms；温恢复 P99 < 3s

### M3：分层归档与生命周期（第 11-13 周）
- 生命周期引擎：HOT→WARM→COLD→RECYCLED 定时流转，TTL 可配
- 冷层转储（Hetzner OS/B2 为主；Glacier IR 仅限极少唤醒）；回填热点 chunk 优先
- **唤醒直方图 pre-warm 调度**（《04》#5）：预测唤醒前拉回本地 + 预解压 WS
- 选择性恢复（Manus 式）：RECYCLED 后仅回灌 artifacts/指定目录
- 成本报表：每沙箱存储占用与层级分布
- **退出标准**：冷归档恢复 < 10s 可交互；归档成本达标

### M4：弹性与生产加固（第 14-16 周）
- Scheduler：多节点 placement（粘性 + bin-packing）、心跳与驱逐；蓝绿升级 SOP（快照版本兼容，《04》§6）
- Gateway：端口转发、WebSocket、路由表；zero-trust egress 代理
- 超售：CPU 3x 起步、balloon 回收；压测 50 并发/节点
- 可观测性：OpenTelemetry + Grafana；宿主加固全集（《04》§5）；安全审计
- **退出标准**：3 节点集群，杀任一 worker 沙箱可异机恢复；G1-G6 全部验收 → **开源 v0.1 发布**

### M5（可选，+4 周）：Agent 原生差异化
- fork/branch/rollback 一等 API（《04》#4/#8）：uffd CoW 内存克隆 + ZFS 层链，checkpoint/rollback 目标 <100ms
- time-travel 调试：每步自动快照 + 任意步 fork 重放
- **退出标准**：单沙箱 fork 出 10 分支并行执行，父实例不停顿

## 4. 团队与分工（2-3 人）

| 角色 | 职责 |
|---|---|
| Infra 负责人 | Firecracker/uffd/ZFS/网络，M0-M2 主力 |
| 后端工程师 | API/Scheduler/生命周期/Gateway，M1/M3/M4 主力 |

参考锚点：Manus 联创估计从零自建需 3-5 名全职工程师数月；EmberVM 通过复用 e2b-dev/infra、BuildBuddy 公开方案与 CodeSandbox 蓝本压缩到 2-3 人 × 4-6 个月。

## 6. 风险登记册（概率 × 影响排序）

| # | 风险 | 等级 | 应对 |
|---|---|---|---|
| 1 | 宿主 NVMe 故障丢持久盘（Fly.io 反复发生） | 高 | pause 写穿 L1 + 运行中定期增量（RPO≤5min）；SLA 只按 RPO 承诺 |
| 2 | 跨 FC/内核版本快照恢复失败（官方定义 unstable） | 高 | 快照 `(fc_ver, kernel_ver)` 标签 + 同构池调度 + 蓝绿升级 SOP（《04》§6） |
| 3 | uffd handler 崩溃 → VM 缺页永久卡死 | 高 | 每 VM 独立进程 + watchdog + 缺页超时强制回收重建 |
| 4 | 同快照多次恢复的熵/token 重复（fork 放大） | 高 | VMGenID + resume hook 刷新；默认一份快照 resume 一次，fork 显式声明 |
| 5 | 控制面 DB 单点 | 中 | 托管 PG 或主从+WAL-G；数据面本地续租不依赖控制面存活 |
| 6 | 宿主 OOM 级联（超售+balloon 失控） | 中 | per-VM memory.max、预留 ≥10%、OOM score 先杀新沙箱而非 handler |
| 7 | conntrack/网络耗尽（失控沙箱致邻居 60% 丢包，论文实证） | 中 | `--notrack` + per-VM 限额 + FC rate limiter |
| 8 | 内网带宽不足导致温恢复超标（10GbE 搬 15GB≈12s） | 中 | 采购锁定 ≥25G；lazy 回填先交互后补全；粘性调度最大化热路径 |
| 9 | NVMe 快照写穿磨损 | 中 | DC 级 NVMe、chunk 去重少写、SMART TBW 监控、系统盘 RAID1 |
| 10 | e2b-dev/infra 上游变动/授权（BSL 部分） | 中 | 只借鉴 orchestrator/template/uffd 设计，Apache 部分并入并署名，控制面自研 |
| 11 | 脏页邻居干扰（93.4% IO 降级，论文实证） | 低 | `vm.dirty_bytes` 调低 + cgroup io.max + FC block limiter |
| 12 | 供应商涨价/缺货（Hetzner 已涨 30-40%、colo 暂停接单） | 中 | 全 Ansible/Nix 化，2 周内可整体迁 OVH/Latitude |
| 13 | 单机房故障 | 低 | 节点分散不同 DC 排；控制面备份异地；区域 DR 是重建剧本非热备 |

## 7. 验收与测试计划

- 性能：恢复延迟分布（P50/P90/P99）× 三级状态 × 数据盘 5/10/20GB 矩阵压测（结果对照《04》§2 预算模型，公开发布）
- 正确性：pause/resume 后进程存活、TCP 重连、时钟、随机数唯一性用例
- 故障注入：kill uffd handler、kill Firecracker、断网、拔节点——验证自动回收与异机恢复
- 成本：30 天生命周期模拟（1000 沙箱），核对各层账单与 G3
- 安全：逃逸基线扫描、egress 绕过测试、快照加密验证
- 开源可部署性：矩阵内每类环境（裸金属/GCP/AWS/DO）跑通 `deploy/singlenode`（《06》§2）
