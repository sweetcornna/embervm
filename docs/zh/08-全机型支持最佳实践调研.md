# 全机型支持最佳实践调研:从 Linux+KVM 到"任何机器"

> 日期:2026-07-10 · 回答的问题:若 EmberVM 要支持所有机器,后续最佳实践应当怎么做
> 调研方法:54 个并行研究/核查智能体——8 个外部维度(dev-VM 工具链、Apple 虚拟化、WSL2、运行时抽象、gVisor、CRIU、libkrun、竞品格局)+ 1 个仓库接缝分析 + 3 轮补漏(安全 delta、VPS 地形与滥用合规、CI/签名/分发);每维度的关键论断由独立智能体做对抗性核查,数条被驳倒并已修正——正文只保留核查后的事实。

## 0. 一句话结论

**"支持所有机器"= 三层,各自有已被业界验证的最佳实践,且都不动 Firecracker 底座(doc 07 结论不变):**

- **(a) macOS/Windows 开发层** —— Docker Desktop 模式:CLI 自动置备一个 Linux VM,在里面跑**原封不动的生产数据面**。macOS 硬门槛 M3+/macOS 15+(嵌套虚拟化),Windows 硬门槛 Win11 x64 + WSL2。这条路 2024-2026 已被 Lima/Colima/Podman 铺平,是跟随而非开荒。
- **(b) 无 KVM Linux 生产层** —— 在现有 `nodeapi.Agent` 接缝后面插入 **gVisor systrap 后端**:它是 2026 年唯一同时满足"无需 /dev/kvm + 有内存态快照 + 有大规模生产验证(Modal/GKE/OpenAI)"的选项;runc+CRIU 被裁决出局(Docker checkpoint 十年 experimental,io_uring 不可 checkpoint)。降级必须**诚实**:能力位 + 类型化 ErrNotSupported + 调度器过滤,Kata 的静默 no-op 是要避开的反模式。
- **(c) 原生 Darwin/Windows 数据面** —— 不做,且有新证据支持:Firecracker 官方拒绝了社区 macOS 移植(快照是牺牲品);所有商业厂商(E2B/Modal/Fly/Vercel/Cloudflare)都是 Linux-server-only。唯一观察项:libkrun 2.0 正在上游合并快照/恢复(设计与 EmberVM 同构),6-12 个月内可能改变 M1/M2 的回退故事。

商业校准:Daytona 2026-06 闭源、E2B 自托管极重且无本地模式、Fly/Morph 把 memory-snapshot 在封闭云端商品化——**"同等快照能力 + 任何机器可自托管"正好是市场空出来的位置**。

## 1. 三层问题定义与市场校准

采纳漏斗分三个门槛,没有一个现有玩家全覆盖:

| 机器类别 | 谁被卡住 | 现状 |
|---|---|---|
| 开发笔记本(macOS/Win) | OSS 第一触点:开发者用"本地几分钟跑起来"评估项目 | E2B 无本地模式(issue #864 明确有需求,官方只回以企业 BYOC);microsandbox 6.9k stars 全靠 laptop-local 定位 |
| 便宜 VPS(无嵌套虚拟化) | 自托管社区(愿意布道的人群) | Hetzner Cloud/Vultr/Linode/Contabo VPS 均无 /dev/kvm;无人提供诚实降级层 |
| 裸金属/嵌套虚拟化云 | 商业生产 | 所有厂商都在这里,已覆盖(doc 06) |

关键市场事实(均经核查):

1. **E2B 自托管是反面教材**:强制 Terraform v1.7.5 + Packer + Nomad/Consul + Cloudflare 账户,仅 GCP(全量)/AWS(beta),裸金属"planned"。第三方一致评价"not a helm install"。EmberVM 的"单机 install.sh + 任意 Linux"要保持在这个复杂度天花板之下并写进定位。
2. **Daytona 闭源腾出了位置**(2026-06,AGPL repo 冻结):"活跃维护的开源沙箱云"空缺。同时 Daytona 证明了**披露的降级商业上可接受**——容器默认档(sysbox-runc,快照不含内存态)拿到 $24M A 轮。注意核查修正:sysbox 是其默认而非可选,且其现在有单独的 VM 类(内存暂停)——先例形态是"容器默认 + VM 强档",与我们"KVM 强档默认 + 容器降级档"正好互补。
3. **快照技术本身在被商品化**:Fly Sprites(2026-01,~300ms checkpoint/restore)、Morph Infinibranch(<250ms 分支)都是封闭云。持久差异化是"同能力但随处自托管",不是快照本身。
4. **Cloudflare 的本地故事是"模拟"不是"移植"**(wrangler dev = workerd 模拟 + 本地 Docker)——证明头部厂商也不做原生跨平台数据面;而我们的 tier (a) 比模拟更强:VM 里跑的是真生产栈,SDK 字节级一致。

## 2. Tier (a)-macOS:自动置备 Linux VM

### 2.1 硬门槛与能力矩阵(全部经核查)

| 条件 | 事实 |
|---|---|
| 嵌套虚拟化 | **M3+ 且 macOS 15+**,Linux guest only,`VZGenericPlatformConfiguration.isNestedVirtualizationSupported`。注意:M2 硬件其实有 FEAT_NV2(Asahi Linux 6.16+ 可用 NV2),M3+ 是 **Apple 的 macOS 软件门槛**,无解释、无放宽迹象(macOS 26 未变)——对我们约束等价于硬件门槛 |
| 嵌套层数 | **只有一层**:vz VM 里 /dev/kvm 可用(Firecracker 够用),但 FC guest 不能再向内暴露虚拟化(lima #4498)。文档化为 dev/prod 语义差异 |
| 架构 | **aarch64 端到端**:x86_64 FC guest 在 Apple Silicon 上不可能(FC FAQ:guest 必须同架构);快照按架构 + CPU 特征绑定,Mac 产的快照永不同步到 x86 生产。Rosetta 只救 VM 内零散 amd64 工具,不在 microVM 数据路径上 |
| 现有 VM 不可借用 | Docker Desktop 与 OrbStack 至今**都不暴露 /dev/kvm**(前者 issue 无回应,后者官方"等 Apple"——而 Apple 已经支持了)。`embervm dev` 必须自己拥有 VM 生命周期,与它们共存 |
| 工具链成熟度 | Lima ≥1.0(`nestedVirtualization: true`,现应 target v2.x——v1.x 已 EOL 2026-02);Colima ≥0.8.0;vfkit `--nested`(2025-06);krunkit `--nested`(v0.2.0);**podman machine 在 M3+ 上已默认开启**——这是铺好的路 |

### 2.2 工程最佳实践(从 Docker Desktop/Lima/Colima/Podman/Finch/Rancher 的事故与设计中提炼)

1. **构建/签名是第一陷阱**:调用 Virtualization.framework 的二进制必须**在 macOS 15 SDK 上编译**并带 `com.apple.security.virtualization` entitlement——Rancher Desktop 带着完整嵌套虚拟化代码但在 macOS 13 上编译,功能静默死亡;Colima 能用只因 Homebrew 逐版本编译。CI 必须有"产物在 M3 runner 上真的出 /dev/kvm"的门禁。ad-hoc 签名够 Homebrew 分发(brew 重签保留 entitlements);直接下载分发需 Developer ID + hardened runtime + notarize。**永不依赖 `com.apple.vm.networking`**(Apple 审批制 entitlement)——学 Lima 用 shared/user-mode 网络。
2. **镜像分发学 podman machine-os**:VM 磁盘镜像发布为 OCI artifact,**tag = embervm 版本**,CLI 与 VM 强制版本握手;digest 固定、验证即失败关闭(Colima #121 教训)。预烘焙一切依赖(colima-core 模式),不做 first-boot cloud-init 安装;残留置备脚本必须幂等,"修复 = 重放"。
3. **系统盘/数据盘分离是不可谈判的**(Docker Desktop 4.44 "容器全没了" #14901、Colima 0.6 升级毁 VM #912、Rancher 双 distro 模式):guest OS 镜像每版本整体替换、不可变;ZFS 池、chunk 缓存、沙箱元数据放**独立数据盘**,升级永不触碰;毁灭性操作只藏在显式的 `embervm dev factory-reset` 后面。
4. **virtiofs 只用于源码共享**:vz 的 virtiofs 有慢性 uid/gid、chmod、mknod 缺陷(lima #2437/#4053,docker/for-mac 多例,本仓库 colima 实测同样踩过 mknod)——ZFS vdev、chunk 缓存、FC workdir 全部放 VM 自己的 raw/ext4 盘。
5. **修复 UX 学 Docker Desktop 分级**:restart → purge VM 数据(保配置)→ factory reset,外加可脚本化的 `embervm dev diagnose`(约 40 项自检 + 支持包)。预检 `embervm dev doctor` 依次报:芯片代(M3+/FEAT_NV2)→ macOS ≥15 → VM 启动后 in-VM /dev/kvm(kvm-ok 等价),每步给独立可行动的错误——这正是 Lima/Colima 用户今天迷路的地方。
6. **socket 转发而非 TCP 端口**:Lima 原生支持 guest unix socket → host unix socket(gRPC 转发器,vz + systemd ≥256 走 vsock)——把 embervm 控制面 socket 以 Docker/podman 的方式呈现在宿主机。
7. **仓库现成资产直接复用**:`deploy/singlenode/install.sh`(一键 Ubuntu 24.04 + loopback ZFS + systemd)在 VM 内原样可用;`scripts/env-check.sh --strict` 就是 in-VM 能力探针;三个 smoke 脚本都已有 Linux guard + self-sudo,VM 内不改即跑。介入点就是 `agent_stub.go` 的 "nodeagent is linux-only" 错误处。

### 2.3 M1/M2 回退与 libkrun 观察项

- M1/M2(或 macOS <15)**不要尝试 KVM 路径**:TCG 全模拟慢 8-12x 不可用。诚实回退:(i) 连远程 Linux worker;(ii) 未来在 plain vz VM 里跑 tier-(b) gVisor 后端(降级标注)。CLI 打印能力矩阵而非在 VM 里迟到地失败。
- **libkrun 快照正在上游化**(这是本次调研最大的变量):RFC #748(2026-06-24)+ PR #762(HVF 上 capture/restore 端到端已工作)+ PR #767(pause/resume),维护者明确想进 2.0;设计(Linux 用 userfaultfd、macOS 用 clonefile CoW、vmgenid reseed)与 EmberVM 架构几乎 1:1。**若合并,原生 macOS 后端(M1+ 全覆盖、无需嵌套虚拟化)从"不可能"变"可行"**——建议跟踪甚至参与上游,但今天不要在它上面建任何东西(v1.19.4 无快照,krunkit API 无 pause 态)。
- Apple 官方 `container` CLI 1.0(2026-06-09)的 `container machine --virtualization`(M3+,需 CONFIG_KVM=y 内核)可作第二置备后端,放在与 Lima 相同的接口后;默认仍 Lima(container 1.0 只支持 macOS 26,挂载/网络成熟度不够)。

## 3. Tier (a)-Windows:WSL2

### 3.1 硬门槛(全部经核查)

- **Windows 11 x64,没有例外**:Win10 上 lifted WSL 无条件写 `ExposeVirtualizationExtensions=false`(#40735,2026-06 仍开着,是对旧 inbox WSL 的回归);WSL-on-ARM64 引导在 EL1,KVM 需要 EL2——**Windows-on-ARM 永久排除**,写进文档防 issue。VDI/云 Windows(双重嵌套)会崩(docker/desktop-feedback #137)——preflight 拒绝。
- `nestedVirtualization=true` 在 Win11 是**默认值**(设置在 `%USERPROFILE%\.wslconfig`,不是 wsl.conf);**stock 内核已带全部所需**(kvm_intel/kvm_amd/vhost_vsock/tun 均 =m)——2020-2023 的"重编内核"教程全部过时,但 =m 意味着 **/dev/kvm 缺失的第一原因是没 modprobe**:发行版启动时 `modprobe kvm-intel||kvm-amd` + udev 权限规则。

### 3.2 打包与运营(核查后的最佳实践)

1. **发行版用 tar-based .wsl 格式**(WSL 2.4.4+):rootfs + `/etc/wsl-distribution.conf`,CLI 经 `wsl --install --from-file` 或 `wsl --import` 程序化安装到自有 distro 名(Docker Desktop 模式);appx/Store 路线自 2024-11 起是 legacy。`/etc/wsl.conf` 开 systemd。
2. **WSL 的三个运营地雷必须设计绕开**:(i) `vmIdleTimeout` 默认 60s——最后一个客户端断开约 1 分钟后**整个 VM 连同所有 microVM 被杀**,node agent 必须以 systemd 服务常驻 + Windows 侧保持连接;(ii) `autoMemoryReclaim` 2026 默认已是 dropCache——空闲触发的页缓存清空与 uffd working-set prefetch 直接对抗,为 embervm distro 建议 disabled 并做 A/B 基准;(iii) 睡眠后时钟漂移上游未修(#10006 megathread)——每次 CLI 连接强制 resync,否则 SigV4(>15min skew)和租约逻辑会炸。
3. **热数据全部放 distro ext4 VHD**,永不放 /mnt/c(9P 慢 2-10x);Windows 侧访问走 `\\wsl.localhost\`。localhost NAT 转发默认可用,不要求 mirrored 模式。
4. **ZFS 在 WSL2 只能自编内核**(kernelModules VHD 也要逐内核版本重编,微软 6.18.x 一月数发)——**dev tier 在 WSL2 用 plain 存储后端**(`pkg/storage` 已有,正是为此准备的降级),不追 ZFS。

## 4. Tier (b)-无 KVM Linux:gVisor systrap 后端

### 4.1 为什么是 gVisor(CRIU 的裁决)

| 候选 | 裁决 | 关键证据 |
|---|---|---|
| runc/Docker + CRIU | **出局**(内存态路线) | docker checkpoint 2016 至今 experimental,多个 2017-2021 restore bug 仍开着;**io_uring 不可 checkpoint**(PR #1597 五年未合,作者退出)且 Node/Postgres 18/Go 生态在普及 io_uring——每 workload 轮盘赌;criu lazy-pages 无生产用户(#2399 会把全部页 fault 进来)。CRIU 本体活跃但动力全在 GPU 批处理(Virtuozzo/Cedana 有生产先例,但都是同构协作负载,非任意多租户沙箱) |
| runc restart-only | **保底档可选** | Daytona 模式:停容器、保文件系统、重执行 entrypoint。诚实、便宜,但只值一个"容器档",不解决暂停/恢复 |
| **gVisor runsc(systrap)** | **采纳** | 无需 /dev/kvm(seccomp SECCOMP_RET_TRAP;ptrace 平台已废弃);checkpoint/restore 有 `--background` 惰性恢复、零页跳过、跨机恢复(CPU 特征 + 版本匹配)、一 checkpoint 恢复 N 份(分支比 FC 还便宜);生产证据:Modal(2025-01 memory snapshots,1B+ 沙箱)、GKE Pod Snapshots(Google 把整条 E2B 式管线产品化)、OpenAI Code Interpreter |

核查修正(重要):(1) runsc checkpoint/restore 官方**仍标 experimental**——"生产成熟"是第三方(Modal/腾讯)实践背书,不是 Google 官方状态,措辞和风险评估要如实;(2) "一 checkpoint→N 副本"是架构推断 + Modal 实践,**不是文档承诺**;(3) 现代 runsc `net-disconnect-ok` 默认 true——**不需要 pre-drain 连接**,save 时自动断开,恢复后连接已关(与 FC resume 语义等价),比初判省一块工程。

### 4.2 工程量与语义边界

- **是编排工作不是内核工作**(Modal 原话感谢 stock gVisor):要建的是 pages file 的 chunk 化惰性服务(复用现有 S3 CAS + working-set prefetch 设计,gVisor 自己会优先 faulted 页)、调度器兼容性 keying(CPU 特征指纹 + runsc 版本 + 架构;gVisor 周发版,**runsc 按快照代 version-pin**)、恢复后 fixup(时钟跳变、新网络身份、应用级 ID/seed 再生)。量级 weeks-to-months,对照 M2 的 uffd 工程是小的。
- **兼容性天花板要写成 API 级合同**:支持——语言运行时、pip/npm、编译器、服务器、docker-in-gvisor(tmpfs /var/lib/docker + iptables=false);不支持/脆弱——systemd、eBPF、完整 cgroup、io_uring(默认关)、内核模块。先例:Cloud Run gen1(gVisor)→gen2(microVM)正是因兼容性回退;Modal 2026 反向推出 VM Sandboxes 补 gVisor 缺口。**我们是在 VM 舰队上加 gVisor 档,方向比 Modal 顺。**

### 4.3 安全 delta 与租户策略(补漏调研结论)

- 边界性质不同:FC = 硬件 VT-x 墙 + ~52K 行 Rust、5 个设备(2018-2025 零逃逸 CVE,2026 年首两例均需非默认配置——PCI transport 保持关闭);gVisor = 用户态 Sentry + ~70 个宿主 syscall 的 seccomp 过滤,**边界被真实跨过过**(CVE-2020-10890、2020 GKE metadata 逃逸等)。
- **SMT 侧信道两者都不防**(gVisor 官方文档明说;FC 同样依赖宿主关 SMT/core scheduling)——而便宜 VPS 租户控制不了宿主微码/SMT。**政策:无 KVM worker 单租户 pinning(一 worker 一信任域);多租户装箱只发生在 KVM worker 上。**这条写进 ADR 与文档的隔离矩阵。
- gVisor rootless 模式官方承认防御纵深更弱——tier (b) 用 rootful runsc + 现代 6.x 内核 + AppArmor/seccomp 分层 + cgroup 资源上限。

### 4.4 VPS 地形与滥用合规(补漏调研结论)

- **按机器类别不按供应商**做门禁:只支持全 VM(KVM/Xen)与裸金属;OpenVZ/LXC 容器 VPS 直接 preflight 失败(无内核模块、gVisor 不可靠)。嵌套虚拟化地形:OVH Public Cloud(官方开 vmx,迁移可能 panic)/Scaleway(社区确认)/DO(能用但官方不支持)有;Hetzner Cloud/Vultr/Linode/Contabo VPS 无——**运行时探测(/dev/kvm open + KVM_GET_API_VERSION),不硬编码供应商**。AWS 2026-02 给 C8i/M8i/R8i 开了嵌套虚拟化,"无 KVM 人群"在缩小但长尾仍大。
- **AUP 合规是功能不是文档**:Hetzner 合同明文禁挖矿与扫描外网,DO/OVH 同类且"租户造成的滥用 = 客户责任"。默认 egress 策略必须与商业玩家收敛的形态一致(E2B/Modal/Daytona/Cloudflare 全部:L3/L4 CIDR + L7 域名过滤 = 443 SNI + 80 Host 头、掉 UDP 443 逼回可检查的 TCP、allow 优先):per-sandbox netns nftables default-drop、53 只准到本地记录型 resolver(关 DNS 隧道/rebinding)、封 25/465/587 与 RFC1918/169.254.169.254、SYN 速率限制、veth 上 tc 带宽帽、挖矿启发式(全核长跑 + stratum 端口/域名)自动杀箱。好消息:这套全部作用在 netns 上,与后端无关——现有 `egress_linux.go` 的 netns 设计直接受益。

## 5. 后端抽象:接口、能力位、错误契约

### 5.1 现有接缝(仓库事实,file:line 经核实)

- 接缝**已经存在且干净**:`nodeapi.Agent` 20 个方法(agent.go:100-143),控制面两种接线(in-process dev / HTTP-unix)都只消费接口;`nodeagent.New(Config)` 就是分发点。方法分类:9 个后端中性(BuildTemplate/Create/Stop/Status/Exec/文件 IO——guestd 是 VM 的 PID 1,容器后端跑同一个 guestd 二进制当 entrypoint 即可**白捡一半接口面**,egress 也作用在 netns 上)、8 个快照依赖(Pause/Resume/Snapshot/ReleaseLocal/Restore/Fork/Rollback)、3 处 FC/ZFS 泄漏(SetBalloon 名字、Prewarm 的 chunk 假设、CreateSandboxRequest 文档写死 128MiB slot/无 vCPU 热插拔)。
- **今天没有任何能力位**:NodeHealth 只有容量字段,调度器只按内存/vCPU 装箱,能力型失败全靠运行时错误字符串(fork_linux.go:199 先例)。这是绿地,没有历史分支要拆。
- 仓库内已有降级先例可照抄:`storage.Backend`(ZFS 全功能 vs plain 诚实降级 + TemplateChecker 可选接口探测)、`Config.RestoreMode`、`JailerBin` 空值。

### 5.2 业界模式与反模式(经核查)

- **反模式(Kata fc.go,已在源码逐行核实)**:PauseVM/SaveVM/ResumeVM 直接 `return nil`,ResizeMemory 返回零值 nil error——**不支持的操作静默"成功"**,上层照常走完,限制只写在文档里。连旗舰级抽象层都这样,"诚实降级"不是可抄的已解问题,是我们要显式设计的。
- 能力表示学 **Nomad**(命名类型字段 struct + 枚举)不学 Kata(匿名 5-bit bitmask);Kata 的正面教训是能力面比想象小(5 bit 撑起 4 个 VMM)且"有可用替代路径的能力用调用方分支,没有的必须报错"。核查修正:Dragonball 不在 Kata 的 Go 接口后面(在 runtime-rs 的 Rust trait 后),Go 接口实际驱动 QEMU/CH/FC/StratoVirt 四个——模式结论不变。
- 调度学 **RuntimeClass**(准入时合并约束 + PodOverhead 按运行时记账),但 K8s 明确把"能力发现"留白了(KEP 原文 out of scope)——**能力对象是我们可以比 K8s 做得更好的点**。心跳上报学 Nomad Fingerprint。
- 错误契约:gRPC **UNIMPLEMENTED**(HTTP 501,"此后端永不支持",SDK 不重试)vs **FAILED_PRECONDITION**(409/412,"当前状态不行");SDK 学 Docker NegotiateAPIVersion——create 响应携带 effective capability set,客户端本地 gate,服务端 501 兜底。**在准入时失败,不在几小时后 pause() 时失败**:create 请求携带所需能力(显式或由模板特性推导),调度器按能力过滤节点。
- 进程拓扑:2-3 个第一方后端**编译进二进制**(containerd/Kata/Nomad 内置驱动同款选择),接口保持 protobuf 可序列化以便未来上 RPC;不引 go-plugin。

### 5.3 设计决定(建议写成 ADR-0008;0007 已被 virtio-mem 预留)

1. `Config.Backend`:`"firecracker"`(默认)| `"gvisor"` | `"runc"`(可选保底档),`New()` 分发到独立 struct(如 `agent_gvisor_linux.go`),各自 `var _ nodeapi.Agent` 断言。控制面零改动。
2. 能力 struct(命名字段 + 枚举):`Pause/MemorySnapshot/DiffSnapshot/LazyRestore/Fork/DataDiskCoW/ResizeAfterBoot bool` + `IsolationClass: microvm|gvisor|container` + `PersistenceClass: memory+disk|disk-only`;NodeHealth 增补 `Backend`、`Caps`、`FCVersion/KernelVersion`(Config 里已有,Healthz 从未上报——跨节点恢复兼容性检查正好需要),全部 additive + omitempty(UpsertNode 零值不覆盖语义友好)。nodes 表加 caps 列,`Scheduler.Place` 加 required-caps 过滤(缺能力的节点等同 state!="up")。
3. `nodeapi.ErrCapabilityUnsupported{Backend, Capability}` 类型化哨兵,HTTP 501 + 机器可读 code;禁止任何后端静默 no-op。
4. **趁 M6 diff 还开着修三处 API 泄漏**:MaxMemoryMiB/MaxVCPUs 文档改为"节点按其后端粒度取整";SetBalloon 文档语义改为"内存回收目标"(方法名保留兼容);Prewarm 在无 chunk 管线的后端定义为 no-op 成功。
5. gVisor 后端 v1 诚实子集:全量实现 9 个中性方法(MaxMemoryMiB/MaxVCPUs 用 cgroup 原生实现,比 FC 还容易)+ ResizeSandbox(纯 cgroup 写);Pause/Resume/Snapshot/Fork 先返回 ErrNotSupported,v2 接 runsc checkpoint(独立能力位,不由后端名暗示)。

## 6. CI 与签名分发(补漏调研结论)

| 路径 | 可行性 | 方案 |
|---|---|---|
| Linux x86-64 数据面 | ✅ 免费 | ubuntu-latest 有 /dev/kvm(一条 udev 规则),现有 CI 不变 |
| Linux arm64 数据面 | ❌ 托管 runner 无 KVM(Azure SKU 限制) | **tier-(b) gVisor 后端顺便成为 arm64 CI 的答案**;要 arm64 KVM 用 Cirrus/Namespace(Apple 硅上跑 Linux VM) |
| macOS `embervm dev` e2e | ❌ **任何托管 macOS runner 永远不行**(一层嵌套 + Linux-guest-only 是 Apple 平台约束;GitHub macOS runner 至今还是 M1) | 每 PR:hosted macos-15/26 只做 build + 签名/公证演练 + 单测;merge/nightly:1-2 台裸金属 M4 自托管 runner——Scaleway Mac mini M4(€0.22/hr,最便宜)或 EC2 mac-m4.metal($1.23/hr 但 24h 最小计费,只当常驻)。**不要买 macOS-VM 型 CI(Cirrus/Orka/Anka)跑这个门禁**——macOS guest 里永远起不了 Lima vz VM |
| Windows WSL2 e2e | ⚠️ 部分 | windows-2025 hosted 可 best-effort(WSL2 已预装,`wsl --install --from-file` + dbus-launch 保活;continue-on-error);KVM-in-WSL2 门禁需自托管(Azure Dadsv5/C8i + Win11);Podman 同样用本地 harness 而非托管 runner——这是行业常态 |

签名:macOS 只签调用 vz 的 helper 二进制(entitlement 见 §2.2);Windows 用 **Azure Trusted Signing $9.99/月**——2024-03 起 EV 证书不再买来即得 SmartScreen 信誉,买 EV 的传统建议已过时;.wsl 产物本身不可 Authenticode,签 CLI/MSI + 内嵌 SHA-256 校验;发 winget manifest + Scoop bucket。

## 7. 分阶段路线图建议

1. **M6 内(顺手,~天)**:修 §5.3-4 的三处 API 泄漏;NodeHealth 加 Backend/Caps/FCVersion 字段(additive);写 ADR-0008 框架。
2. **M7a `embervm dev` for macOS(~2-4 周)**:拦截 agent_stub 错误点 → Lima v2(vendored, 版本锁定)置备 vz + nested VM → OCI artifact 分发 guest 镜像(tag=版本)→ VM 内跑 install.sh + env-check.sh → socket 转发 + doctor + 分级修复。CI 上裸金属 M4 nightly 门禁。**这是采纳漏斗第一门槛,优先级最高,且零核心改动。**
3. **M7b Windows WSL2(~1-2 周,复用 M7a 骨架)**:.wsl distro 打包 + .wslconfig 写入/校验 + modprobe/udev + 保活服务 + 时钟 resync;plain 存储后端。
4. **M8 gVisor tier-(b) v1(~2-4 周)**:Backend 分发 + 能力位 + ErrNotSupported + 调度器过滤 + 9 中性方法 + preflight 探针(§4.4 清单)+ 统一 egress 策略移植验证。v1 明确 restart-only。
5. **M8+ gVisor checkpoint 档(~1-2 月,按需启动)**:runsc checkpoint → chunk store → `--background` 恢复 + CPU 指纹 keying + runsc version-pin。启动条件:tier-(b) 有真实用户要求暂停/恢复。
6. **持续观察**:libkrun 2.0 快照(#748/#762)、Firecracker PCI CVE 面(保持 MMIO)、AWS 嵌套虚拟化机型扩散、WSL/GitHub runner 能力变化。

## 8. 反方条件(何时重议本结论)

- **libkrun 2.0 合并快照/恢复且 clonefile CoW 恢复达到可用质量** → 重估"原生 macOS 后端"(M1/M2 全覆盖 + 无嵌套门槛),tier (a) 的 Lima 方案降为其中一个 provider;
- **gVisor checkpoint/restore 官方去掉 experimental 标签或 containerd shim 集成(#12280)落地** → tier-(b) 快照档提前;
- **托管 CI(GitHub macOS runner)获得 M3+ 硬件或嵌套虚拟化** → 裸金属 CI 预算取消(目前无任何路线图迹象);
- **嵌套虚拟化在主流廉价 VPS 普及**(AWS C8i 是开端) → tier-(b) 收缩为纯 CI/arm64 用途,投入降级;
- **GPU 沙箱进入路线图** → gVisor 后端从"降级档"升为"GPU 档"候选(nvproxy + cuda-checkpoint 已可用),与 doc 07 §5 反方条件合流。
