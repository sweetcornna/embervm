# 沙箱云服务调研报告

> 项目：EmberVM（余烬）开源沙箱云
> 日期：2026-07-03
> 目标：调研 E2B、Manus、Cursor 云等沙箱基础设施的原理，为自建"持久化存储 + 弹性计算 + 分层冷归档 + 秒级恢复 10-20GB 数据"的沙箱云服务提供技术依据。
> 本文为行业调研记录；具体选型以《02-技术架构设计》与《04-创新与最佳实践》§1 数据验证结论为准。

---

## 1. 调研范围与核心问题

自建沙箱云需要回答四个问题：

1. **隔离与运行时**：用什么技术跑不可信代码（Docker / gVisor / Kata / Firecracker）？
2. **秒级恢复**：10-20GB 的沙箱状态（内存 + 磁盘）如何在秒级内恢复？
3. **持久化与分层归档**：沙箱休眠后数据放哪里、成本如何压到最低？
4. **弹性计算**：如何在节点池上调度、超售、扩缩容？

行业内 E2B、Manus、Cursor、Modal、Morph、CodeSandbox、Fly.io、Cloudflare 等已给出高度趋同的答案，下文逐一拆解。

## 2. E2B 架构详解（最重要的开源参照）

E2B（github.com/e2b-dev/infra，Apache-2.0）是目前唯一完整开源的生产级 Firecracker 沙箱云，**Manus 的底座就是自托管的 E2B**，是本项目最直接的参考实现。

### 2.1 总体架构

```
Client → Client-Proxy（边缘路由，Consul 服务发现 + Redis 路由状态）
       → API Server（REST/Gin，鉴权）
       → Orchestrator（gRPC，单机 Firecracker 编排，需 root）
       → Firecracker microVM
       → envd（VM 内 daemon，Connect RPC，进程/文件 API，端口 49983）
```

- 数据面：PostgreSQL（主数据）、Redis（缓存/状态）、ClickHouse（指标分析）
- 基础设施：Terraform + HashiCorp Nomad（作业调度）+ Consul（服务发现），部署在 GCP
- 模板产物存 GCS bucket；观测栈 OpenTelemetry + Grafana LGTM

### 2.2 模板构建：Docker 镜像 → microVM 快照

关键洞察：**E2B 沙箱的"创建"本质是从预构建快照 resume，而不是冷启动内核**——这是其宣称 ~150ms 启动的来源。

构建流水线（BaseBuilder → UserBuilder → StepBuilder → PostProcessing）：

1. 从 Registry 拉 OCI 镜像，抽取 layers 直接转成 **ext4 块设备镜像**
2. 用临时 BusyBox init 启 VM 跑 provision.sh（装 systemd、SSH、chrony 等）
3. Dockerfile 风格的每个 RUN/COPY 步骤在真实沙箱内经 envd 执行，然后 `PauseAndUpload` 冻结保存 diff——**每层本身就是一个 Firecracker 快照层**（内容寻址缓存）
4. 最终按目标 CPU/RAM 重启、跑 StartCmd、等 ReadyCmd、刷盘、拍最终快照

模板产物三件套：`rootfs.ext4`（文件系统）+ `snapfile`（vCPU/设备状态）+ `memfile`（内存快照）。

### 2.3 运行时与快照机制

- 每个 Firecracker 进程：独立 mount namespace（`unshare -m`）+ 独立 cgroup + 独立 netns；rootfs 是 NBD 承载的 CoW 文件
- rootfs 采用 overlayfs 方案：只读 base（squashfs）+ 每实例可写 ext4 sparse overlay（`init=/sbin/overlay-init`），避免每实例复制几百 MB
- **Resume 路径**：`LoadSnapshot` 用 uffd（userfaultfd）memory backend，orchestrator 自带缺页 handler，按需拉页
- **差分快照**：pause 时从 Firecracker 取 dirty-page bitmap，只写脏页 diff；rootfs 修改也按 diff 存；跨节点 resume 走 P2P chunk 传输，避免全量从 GCS 拉

### 2.4 官方持久化数字

| 指标 | 数值 |
|---|---|
| 沙箱创建（模板快照恢复） | ~150ms |
| pause | ≈ 4 秒 / GiB RAM |
| resume | ≈ 1 秒 |
| paused 沙箱保留 | 无限期（需显式 kill） |
| 连续运行上限 | Pro 24h / Hobby 1h |
| 默认规格 | 2 vCPU / 512 MiB，上限约 8 vCPU / 8 GiB |
| 计费 | vCPU ~$0.05/h，RAM ~$0.018/GiB·h，存储免费 |

## 3. Manus 原理

### 3.1 沙箱模型

- 每个任务分配一台**完全隔离的云端 Linux VM**（自带浏览器、shell、Python、文件系统），可 24/7 运行
- 生命周期三段式：**Create → Sleep/Awake →Recycle/Recreate**
  - 无操作自动休眠，回到任务自动唤醒，休眠期间文件不变
  - 持续休眠会被回收：Free 7 天 / Pro 21 天
  - 回收后重开任务自动新建沙箱并**选择性恢复**：artifacts、用户上传附件、项目文件自动回灌；中间代码与临时文件丢弃
- Zero Trust 模型：沙箱内给 root、随便折腾，但沙箱无法访问会话/账号数据；不可恢复错误时自动换新沙箱续跑

### 3.2 基础设施：自托管 E2B

E2B 官方 case study 确认：**Manus 跑在自托管的 E2B 上（部署在 Manus 自己的机器上），底层 Firecracker microVM，~150ms 启动**。

选型过程很有参考价值：Manus 先试了 Docker——**容器 spawn 需 10-20 秒**，且不具备完整 OS 能力（无法装系统级包），故弃用。联合创始人估计自建等价设施需 3-5 名全职 infra 工程师数月工作。pause/resume（含内存）被用于等待用户提供凭证、过人机验证等场景；付费用户数据在沙箱中保留最长 14 天。

### 3.3 Agent 层（与 infra 解耦）

- **CodeAct 范式**：模型动作是可执行代码而非 JSON function calling
- 多 Agent：Planner → Executor → Knowledge
- 2025-03 被"扒沙箱"事件：用户直接让 Manus 打包 `/opt/.manus/` 拿到 runtime 与 prompt；联创回应"每个用户本来就能访问自己的沙箱，属设计使然"——印证其安全边界画在沙箱外而非沙箱内
- 工程经验（官方博客《Context Engineering for AI Agents》）：KV-cache 命中率优先、用文件系统作为终极上下文、todo.md 复诵等

## 4. Cursor 云原理

### 4.1 Anyrun 编排层

- **Anyrun**：Cursor 自研编排服务，纯 Rust，负责在云端启动 agent；底层 **AWS EC2 + Firecracker microVM**
- 每个 agent 任务一台独立 Ubuntu VM；CPU 在 AWS、推理在 Azure，Terraform 管理

### 4.2 Workspace 快照机制

启动流程：**VM 从 disk snapshot 恢复 → clone 对应 branch → 执行 environment lifecycle**：

- `.cursor/environment.json` 定义环境，`install`（跑一次后拍快照缓存）→ `start`（每次 boot 执行，取 secrets/起服务）
- 官方博客明确列出自建的核心能力：**消息间隙高效 hibernate/resume VM、快速持久地 checkpoint/restore/fork VM images 的 pipeline、readonly VM、prewarmed VM 池**

### 4.3 控制面：Temporal durable execution

- 初版 work-stealing 架构可靠性只有 1 个 9；迁移到 **Temporal** 后越过 2 个 9，现每天 5000 万+ actions、700 万+ workflows
- **关键设计：Agent loop、机器状态、会话状态三者解耦**——agent loop 活在 Temporal 而非 VM 内；会话是 append-only 存储 + 流式推送。VM 是可丢弃的躯体，编排状态在外部持久层
- 内部 40%+ PR 由 cloud agents 产出；企业可 self-host agent VM（BYOC），Cursor 只管编排

## 5. 竞品格局与横向对比

| 平台 | 隔离技术 | 冷启动/恢复 | 持久化模型 | Snapshot/分支 | 定价 |
|---|---|---|---|---|---|
| E2B（Manus 底座） | Firecracker | ~150ms 创建 / ~1s resume | pause/resume（fs+内存） | 快照 + diff | $150/月 + ~$0.05/vCPU·h |
| Modal | gVisor | 300-500ms；mem snapshot 快 2.5x | fs/dir/memory snapshot + Volumes（7 天过期） | 有 | ~$0.14/vCPU·h（3x 溢价） |
| Morph Cloud | 完整 VM | **<250ms 活体分支** | 不可变快照（内存+磁盘+进程） | 最强：Infinibranch 不停父实例 | usage-based |
| Daytona | Docker（可选 Kata） | <90ms（共享内核） | stateful snapshot | 有 | ~$0.05/vCPU·h，AGPL 开源 |
| Fly Machines/Sprites | Firecracker | resume 数百 ms | 整机 suspend；热 $0.50 / 冷 $0.02 per GB·月 | 有（绑定代码版本） | $0.07/CPU·h + rootfs $0.15/GB·月 |
| Cloudflare Sandboxes | 容器 + Durable Object | snapshot 恢复 ~2s（vs 冷建 30s） | 按 ID 寻址、idle 即眠 | 全盘 snapshot + fork | active-CPU $0.00002/vCPU·s |
| CodeSandbox | Firecracker | **resume 平均 400ms，P99 2s** | 5 分钟不活跃即 hibernate | uffd fork <1.5s | — |
| Blaxel | microVM | resume <25ms | 永久 standby 零计费 | 有 | contact sales |

### 隔离技术选型结论

| 技术 | 启动 | 隔离强度 | 快照 | 适用 |
|---|---|---|---|---|
| Docker/runc | ~50ms | 弱（共享内核） | 无原生（CRIU 脆弱） | 受信内部代码 |
| gVisor | ~百 ms | 中（用户态内核） | 依赖平台自研 | 容器兼容 + 中隔离（Modal） |
| Kata | 秒级 | 强 | 有 | K8s 原生 + 硬隔离 |
| **Firecracker** | 100-200ms | 强（硬件虚拟化） | **原生 snapshot/restore/diff/uffd** | serverless 高密度（E2B/Cursor/Fly/Lambda） |

**结论：跑不可信 AI 代码 + 需要快照原语 → Firecracker 是行业统一答案。** 唯一硬性要求：KVM（裸金属或嵌套虚拟化）。

## 6. 秒级恢复 10-20GB：关键技术拆解

### 6.1 裸吞吐数学（为什么不能整份复制）

搬运 15GB 所需时间：

| 通道 | 吞吐 | 耗时 |
|---|---|---|
| NVMe 本地 | 3-7 GB/s | 2.1-5s |
| 10GbE | 1.25 GB/s | **12s** |
| 25GbE | 3.125 GB/s | 4.8s |
| 100GbE | 12.5 GB/s | 1.2s |

**任何"整份复制"的设计在 10GbE 下都注定 >10s。秒级恢复的正解 = 把"恢复"变成 O(1) 元数据操作 + 按需取数。**

### 6.2 四大核心技术

**(1) CoW clone —— O(1) 建盘**
ZFS clone、Ceph `rbd clone`、dm-thin snapshot 都是毫秒级元数据操作。数据已在池/集群上时，"恢复磁盘"瞬时完成，数据按需读取。

**(2) uffd lazy loading —— 先启动后取内存页**
Firecracker 官方支持 userfaultfd memory backend：VM 立即 resume，缺页时由用户态 handler 按页 `UFFDIO_COPY` 回填。实测可把 restore 从 ~7s 降到 <100ms。磁盘天然 lazy（块设备挂载瞬时），真正需要工程化的只有内存。

**(3) 块级增量（diff）+ chunk 化 —— 只搬变化**
- E2B：dirty-page bitmap → 只存脏页 diff；跨节点 P2P chunk 传输
- Lambda SnapStart：512KB chunk + 三级缓存（本地 → 分布式 L2 → S3）+ 基于 provenance 的跨函数 dedupe，数据搬运减少可达 90%
- CodeSandbox：8KiB chunk + lz4 分块压缩（8GiB 快照 → 745MiB）+ 全零 chunk 跳过 + balloon Free Page Reporting（16GiB 快照中 13.1GiB 直接不存）→ **resume 平均 400ms，P99 2s，250 万次 resume/月**

**(4) 三级恢复分层**
- 热：数据在本机 NVMe，ZFS clone + uffd，<1s
- 温：数据在集群/对象存储，内存 lazy load + 磁盘块按需回填，1-3s
- 冷：从归档层回填，秒-分钟级，靠 prefetch + 并行拉取压时间

### 6.3 必须处理的正确性问题

- 每 clone 独立 netns + NAT（同 guest IP 冲突）
- resume 后 wall-clock 校正（kvm-clock `clock_realtime` / chrony）
- 熵与唯一性：同一快照多次恢复不安全（token/随机数重复）；VMGenID 设备 + 应用层刷新
- uffd handler 崩溃 → VM 永久挂起，需监控回收
- cgroups v2（v1 下 restore 延迟显著劣化）
- 快照文件加密与完整性（Firecracker 只做 CRC）
- CRIU（docker checkpoint）路线：TCP/GPU/挂载一致性限制多，**不建议作为主路径**

## 7. 存储层调研

### 7.1 本地 CoW 存储

| 技术 | 快照 | 特点 |
|---|---|---|
| overlayfs | 无块级快照 | 文件级 CoW，适合只读 rootfs 基础层；首写大文件整文件 copy-up |
| dm-thin | 毫秒级 | 块级 CoW；Docker driver 已 deprecated 但 dm-thin 本身仍常用 |
| **ZFS** | **瞬时（元数据）** | clone O(1)；`zfs send -i` 块级增量；lz4/zstd 压缩（代码类 1.5-2x）；dedup 慎用（OpenZFS 2.3 Fast Dedup 改善但社区仍建议优先 clone 共享 + 压缩） |
| Btrfs | 瞬时 | 小文件写性能差、维护多，小众 |

**最优组合：rootfs 只读层用 overlayfs/erofs，可写持久层用 ZFS。**

### 7.2 分布式/网络存储

- **Ceph RBD**：`rbd clone` 即时（纯元数据）+ copy-on-write/copy-on-read 惰性拉取；全 NVMe 集群 4K 写延迟 0.5-0.73ms；**运维负担重，≥3 节点起步，小团队慎选**
- **JuiceFS**：S3 存 4MB 数据块 + Redis/TiKV 元数据 + 本地 NVMe 缓存；顺序读可达 2GB/s；随机读冷 ~29ms → 缓存命中 0.3ms；适合 `/workspace` 持久目录，不适合当低延迟系统盘
- **自建 S3 底座**：MinIO 已于 2026-04-25 正式 archived（无安全补丁，勿选）；可选 Garage v2.3（轻量，无 versioning/object-lock）或 SeaweedFS（S3 API 更全，含 lifecycle），或直接用 Hetzner Object Storage（Ceph 底座）
- **ublk**（Linux 6.0+，io_uring 用户态块设备）：实现"本地空盘 + 按需从对象存储拉块"的 lazy 网络 CoW 盘，IOPS 约为 qemu-nbd 2 倍

### 7.3 分层冷归档

S3 存储级别（us-east 参考价）：

| 级别 | $/GB/月 | 取回延迟 | 最短存期 |
|---|---|---|---|
| Standard | 0.023 | ms | — |
| Standard-IA | 0.0125 | ms | 30 天 |
| **Glacier Instant Retrieval** | **0.004** | **ms** | 90 天 |
| Deep Archive | 0.00099 | 12-48 小时 | 180 天 |

- Glacier IR 毫秒取回、约 1/6 Standard 价格，但**取回费 $0.03/GB（20GB 唤醒一次 $0.60）——仅适合极少唤醒的沙箱**；高频唤醒场景选无取回费的 Hetzner OS/B2；Deep Archive 只适合确认放弃的沙箱
- 自建：对象存储 lifecycle 分层（本地 NVMe → 自建 S3 → 公有云冷层）
- 内容寻址去重：FastCDC 分块（约 10x 于 Rabin 速度，去重率相当）；同 base image 派生的沙箱间 chunk 去重收益极高（base 层近 100% 共享）；典型去重率 60-80%
- **警告**：restic 类内容寻址仓库恢复吞吐仅 ~80MB/s 量级，只能做归档层，不能上秒级恢复路径

### 7.4 弹性计算

- Firecracker 需 KVM：裸金属或嵌套虚拟化；单 VM 内存开销 <5MiB，单机可上千 microVM
- 云选择：**Hetzner 裸金属最具性价比**（2026-06 涨价后 Auction 翻新机 €25-40/月仍最低）；OVH/Scaleway 同定位；GCP/Azure/DO 支持嵌套虚拟化（有性能折损）；AWS 2026-02 起 Intel 虚拟实例开放嵌套（c8i.large ~$68/月）——逐厂商核实矩阵见《06-云服务器实测指南》
- 超售：CPU 3-10x；内存配合 virtio-balloon 1.5-2x（需 swap/驱逐兜底）
- 本地 NVMe 缓存容量 ≈ 活跃沙箱数 × 平均工作集 + 30% 预留

## 8. 行业共同架构模式（关键结论）

1. **VM snapshot/restore 是全行业核心原语**。E2B、Cursor、Fly、Modal、Morph、Cloudflare、CodeSandbox 全部收敛到：把计算的瞬时状态物化为可存储对象 → 秒级恢复 + 计费停表。快照 fork 正从恢复手段升级为并行探索原语（RL rollout、tree-of-thought）。
2. **秒级恢复的瓶颈不在 CPU 状态，在数据搬运**。所有成功案例的答案一致：lazy loading（mmap/uffd）+ 只搬真正被访问的页 + chunk 级 diff/dedupe/压缩 + 多级缓存。
3. **磁盘与内存分开处理**：只读 base 镜像共享 + 每实例薄 CoW 层；10-20GB 用户数据放数据盘，块设备挂载瞬时、数据按需读——**大数据盘天然不在 resume 关键路径上**。
4. **分层存储 + 选择性恢复**：运行态（热）→ 休眠态（快照，磁盘价计费）→ 回收态（只回灌重要文件）。快照普遍有 TTL（Modal 7 天、E2B 14 天、Manus 7/21 天）。
5. **控制面与沙箱彻底解耦**：沙箱是可丢弃的躯体，会话/编排状态在外部持久层（Cursor 用 Temporal、Cloudflare 用 Durable Object）。
6. **安全边界画在网络层**：沙箱内给 root，egress 代理注入凭证、secret redaction、network policy 管控。
7. **自建 vs 采购**：Manus 选"采购但自托管"（半天接入 vs 自建 3-5 人月级）；Cursor 因环境即产品选全自建。**本项目自建时应最大化复用 E2B 开源代码与 CodeSandbox 已验证的方案。**
