## 一、一句话定位

**EmberVM 是一朵可以自托管的沙箱云:每个沙箱是一台 Firecracker microVM,能在数百毫秒内暂停、休眠、跨节点唤醒,数据零丢失,休眠成本趋近于零。**

名字即隐喻:余烬(Ember)——火熄了,但热还在,一口气就能吹旺。沙箱的常态不是运行,而是休眠;这个系统的全部设计都围绕"如何让休眠足够便宜、唤醒足够快"展开。

面向的场景是 AI Agent 与云端开发环境:成千上万个环境,绝大多数时间处于闲置,偶尔被唤醒执行几分钟。按"常驻虚拟机"的方式经营这种负载,成本是灾难;按"每次冷启动重建"的方式,体验是灾难。EmberVM 选第三条路:**把沙箱变成可以随时熄灭、随时复燃的余烬。**

## 二、四个核心洞察

整个系统建立在四个想法上,其余一切都是推论。

**1. 沙箱是可丢弃的躯体,状态才是本体。**
所有编排状态住在 PostgreSQL,所有数据状态住在快照。任何一台 worker 死掉,上面的沙箱躯体随之湮灭,但本体无损——从最近一次写穿的快照在任意节点复活。这条原则同时解决了容灾、迁移、扩缩容三个问题:它们全都退化为"换个地方 resume"。

**2. 恢复的本质是元数据操作,不是数据搬运。**
唤醒一个 8GiB 内存、20GB 数据盘的沙箱,不需要搬运 28GB:磁盘天然按需读(块设备 lazy);内存用 userfaultfd 懒加载 + 工作集(WS)预取——上次唤醒实际摸过的页(约 9%)优先拉回,其余缺页时再取。于是恢复延迟与沙箱大小**解耦**,只与工作集大小相关。这是"热恢复 P50 < 500ms"敢写进目标的原因。

**3. 存储是温度分层的,成本随温度指数下降。**
HOT(本机 NVMe)→ WARM(对象存储)→ COLD(廉价冷层)→ RECYCLED(仅存 artifacts),TTL 驱动自动流转。配合内容寻址 chunk 去重(同模板派生的沙箱共享 60%+ 的块)与 synthetic-full 合成(只动元数据,不重写数据),M3 实测归档成本为逻辑数据量的 **6.8%**。休眠一万个沙箱的代价,约等于常驻几百个。

**4. 简单的组件 + 严格的契约,胜过聪明的分布式系统。**
不自研状态同步(Fly.io Corrosion 的教训),不引入 Nomad/Consul/Redis 集群:PostgreSQL 是唯一事实源,沙箱行里的 `node_id` 就是路由表,调度器靠轮询心跳判活。所有并发正确性收敛为一条纪律:**状态转换必须 CAS——破坏性操作先赢得状态,再动手;复制型操作先备好副本,再 CAS,再清理。**

## 三、架构:三个平面

```
控制面   Gateway → API Server(Gin) → Scheduler → 生命周期引擎
         └────────────── PostgreSQL(唯一事实源)──────────────┘
                              │ nodeapi (HTTP/unix socket)
数据面   Node Agent(root,每节点一个)
         ├─ Firecracker ×N ← jailer(chroot + 独立 uid/gid + netns + seccomp)
         ├─ uffd handler ×N(每 VM 一进程:懒加载 + WS 预取)
         ├─ guestd(VM 内 daemon:exec / 文件 / 健康 / resume hook)
         ├─ watchdog(僵尸收割:进程暴毙 → FAILED → 异机可复活)
         └─ ZFS 池(dataset + raw file,clone O(1),send -i 增量)
                              │ chunk 推拉
存储面   L1 对象存储(S3 兼容,FastCDC 内容寻址 chunk 仓库)
         L2 冷层(synthetic-full 合成 + mark-and-sweep GC)
```

关键设计选择及其"为什么":

| 选择 | 为什么 |
|---|---|
| Firecracker 而非 Docker/gVisor | 硬件隔离 + 原生 snapshot/diff/uffd,行业统一答案;Docker spawn 10-20s 且无完整 OS |
| ZFS dataset + raw file,**不用 zvol** | clone 毫秒级建盘;zvol 实测慢 25%~6 倍且有快照空间陷阱 |
| uffd + WS 预取**必选**,纯 lazy 是陷阱 | 纯懒加载吞吐 43MB/s,预取 533MB/s——差一个数量级 |
| 数据盘与内存分开处理 | 10-20GB 数据盘不进 resume 关键路径,这是"大盘一等公民"的前提 |
| jailer chroot 内路径恒为 /data、/snap | 快照里记录的路径在**每个节点都相同**——跨节点恢复的路径稳定性是构造出来的,不是修补出来的 |
| PostgreSQL 单一事实源,无 Redis/gossip | 少一个组件就少一类故障;心跳轮询对百节点规模绰绰有余 |
| 安全边界画在 microVM + 网络层 | 沙箱内直接给 root,能力完整;租户间无东西向网络 |

## 四、生命周期:一台状态机统治一切

```
CREATE → RUNNING ⇄ PAUSED_HOT → PAUSED_WARM → ARCHIVED_COLD → RECYCLED
             │         (TTL)        (TTL)          (TTL)      (仅 artifacts,
           FAILED ──resume──▶ 任意节点复活                      可选择性恢复)
```

- 状态转换的合法边收敛在一个包(`pkg/lifecycle`)里,控制面与节点对"什么是合法"永远一致;
- 每次 pause 立即**写穿 L1**(RPO ≤ 5min):这一条让 FAILED 不是终态——节点死亡、进程僵死,都只是"从最近快照异机 resume"的普通路径;
- 唤醒预测(Serverless-in-the-Wild 直方图)在预测窗口内**预热**:提前把 WS chunk 拉回本地,冷唤醒被削成温唤醒。

三级恢复的实测答卷(CI 全栈 e2e,真 microVM):热 P50 **141ms**、温 P99 **1.25s**、冷 **453ms**(门槛 <10s)、跨节点 **1.63s**。

## 五、工程方法论:这个项目如何保证自己是对的

这部分与架构同等重要——**本地开发机(macOS)没有 KVM,GitHub Actions 是唯一能跑真 Firecracker 的环境**,于是 CI 被迫成为第一性的工程实践:

1. **每个里程碑一个 e2e 退出门禁**(e2e-m1/m2/m3……):真实 REST + 运行中的引擎 + ZFS + 双对象存储 + PostgreSQL,门槛数字(P50/P99/成本比)写进 workflow,用 `--- PASS` 逐条 grep——跳过的测试永远不能伪装成通过。
2. **带着运行中引擎的 e2e 暴露同步测试永远发现不了的竞态**——M3 的 CAS-before-copy 窗口在 e2e 首跑几秒内现形。事后所有破坏性/复制型转换纪律皆源于此。
3. **教训显式沉淀**:每个非显然的坑(僵尸进程应答 signal-0、jailer 按 exec-file basename 建 chroot、pgx 列数漂移是运行时错误……)都进 ADR 或提交信息,下次不再交学费。
4. **深度复用但只抄对的**:e2b-dev/infra 的 Apache 部分可并入(AGPL 兼容),BSL 部分只读不抄;其官方承认未解决的网络隔离/LB/部署三件事,恰是自研重点。

## 六、走到哪了:里程碑与实测数字

| 里程碑 | 交付 | 关键实测 |
|---|---|---|
| **M0** | 环境与原型验证、基线方法论(ADR-0001) | — |
| **M1** `v0.1.0-m1` | 单机 MVP:REST 全生命周期、netns 池、guestd | 20 并发,热恢复 71ms |
| **M2** `v0.2.0-m2` | chunk 化恢复管道:uffd + WS 预取 + L1 写穿(ADR-0003) | 热 P50 141ms,温 P99 1.25s,跨节点 1.63s,去重 61.8% |
| **M3** `v0.3.0-m3` | 四级归档与生命周期引擎、synthetic-full、GC、预热预测(ADR-0004) | 冷恢复 453ms,归档成本比 6.8% |
| **M4** 进行中 | P1 多节点调度器 ✅ P2 jailer 加固 ✅ P3 僵尸看门狗 + balloon 管路 ✅ | 全部 CI 验证 |

**M4 剩余 → 开源 v0.1 发布**:超卖策略(CPU 3x + balloon 回收)、Gateway(端口转发/WebSocket/路由)、可观测性(OTel + Grafana)、单节点 50 并发压测、3 节点集群 e2e(杀任一 worker → 沙箱异机恢复)、G1-G6 全项验收。

**G1-G6 验收目标**(《03》§1):秒级恢复 / 数据零丢失 / 冷层 ≤$5/TB/月 / 创建 <500ms 且 ≥50 并发/节点 / 僵死自动回收 / 一台云 VM 跑通 `embervm dev` 完整体验。

**M5(可选)**:fork/branch/rollback 一等 API——uffd CoW 内存克隆 + ZFS 层链,单沙箱 fork 十个分支并行执行。这是对"Agent 探索多条路径"场景的原生回答,也是与所有现有开源方案的差异化所在。

## 七、一段话收束

EmberVM 押注的是一个正在到来的负载形态:**海量、长寿、绝大部分时间沉睡的计算环境**。它的回答由三层组成——用 microVM 换隔离,用"快照即本体"换容灾与迁移,用温度分层与内容寻址换成本;再用一条 CAS 纪律和一套真 microVM 的 CI 门禁,保证这一切在并发与故障面前依然成立。火可以随时熄,余烬常温热。
