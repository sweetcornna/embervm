# 技术架构设计：EmberVM（余烬）沙箱云

> 依据《01-调研报告》结论设计，选型均经《04-创新与最佳实践》§1 数据验证。
> 核心指标：热恢复 P50 < 500ms；温恢复（含 10-20GB 数据盘）P99 < 3s 可交互；持久数据 RPO ≤ 5min；冷归档 ≤ $5/TB/月。

---

## 1. 总体架构

```
                        ┌─────────────── Control Plane ───────────────┐
  Client/SDK ──HTTPS──▶ │ Gateway/Proxy ─▶ API Server ─▶ Scheduler    │
                        │        │             │            │         │
                        │      Redis       PostgreSQL   (placement,   │
                        │   (路由/状态)    (元数据主库)   bin-packing) │
                        └──────────────────────┬──────────────────────┘
                                               │ gRPC
        ┌────────────── Worker Node (裸金属, KVM) ────────────────┐
        │ Node Agent (orchestrator)                               │
        │  ├─ Firecracker microVM ×N ── guestd (VM 内 daemon)     │
        │  ├─ uffd handler ×N (每 VM 一进程：懒加载 + WS 预取)     │
        │  ├─ 本地 NVMe ZFS 池 (L0 热层: rootfs + 数据盘)          │
        │  └─ netns/veth/NAT per sandbox                          │
        └───────────────────────┬─────────────────────────────────┘
                                │ chunk 拉取 / 归档推送
        ┌───────────────────────▼─────────────────────────────────┐
        │ Storage Plane                                           │
        │  L1: Garage/SeaweedFS 或 Hetzner OS ← FastCDC chunk 仓库 │
        │  L2: 低价冷归档 (lifecycle 转储；Glacier IR 仅限极少唤醒) │
        └─────────────────────────────────────────────────────────┘
```

设计原则：

1. 沙箱是可丢弃的躯体，所有编排/会话状态在控制面持久层
2. 恢复 = O(1) 元数据操作 + lazy 取数 + WS 预取，绝不整份复制
3. 磁盘与内存分开处理：数据盘不进 resume 关键路径
4. 安全边界画在 microVM + 网络层，沙箱内给 root
5. **零厂商锁定**（开源自托管定位）：任意 S3 兼容存储、PostgreSQL+Redis、无 Nomad/Consul/GCP 依赖；`embervm dev` 单进程模式一台云服务器即可完整体验（见《05-开源项目规划》）

## 2. 组件设计

### 2.1 控制面

| 组件 | 职责 | 选型 |
|---|---|---|
| Gateway/Proxy | 沙箱端口转发、WebSocket、路由到 worker | Go + Redis 路由表（参考 e2b client-proxy） |
| API Server | REST API：create/pause/resume/snapshot/kill、鉴权、配额 | Go/Gin + PostgreSQL |
| Scheduler | placement（优先回原节点）、bin-packing（内存为主要约束）、扩缩容 | 自研；控制面 DB 为 source of truth + 内存态集群视图；**不自研分布式状态同步**（Fly.io Corrosion 教训，《04》§6） |
| 生命周期引擎 | 沙箱状态机、TTL、归档/回收、唤醒直方图预热 | 嵌入 API Server；后期可上 Temporal |

### 2.2 数据面（Worker Node）

| 组件 | 职责 |
|---|---|
| Node Agent | 节点唯一常驻服务（需 root）：管理 Firecracker 进程、netns、cgroup v2、ZFS 卷、快照上传下载、心跳；控制面失联时本地续租，沙箱继续存活 |
| Firecracker | microVM 运行时；jailer 加固；virtio-balloon + virtio-rng + VMGenID；io_engine=Sync（Async 仍 dev preview，跟进 v1.13+ PCIe transport） |
| uffd handler | **每 VM 独立进程**：chunk 化 memfile（8-16KiB + lz4 + 零页跳过）懒加载、WS trace 记录与顺序预取（REAP/FaaSnap 式，必选）、watchdog 看护（handler 挂 = 缺页永久卡死） |
| guestd | VM 内 daemon（参考 envd）：进程执行、文件读写、端口发现、健康检查、resume hook（校时/token 刷新） |

### 2.3 存储引擎

**每个沙箱两块盘：**

- **系统盘**：只读 base 镜像（squashfs，模板共享）+ 可写 overlay（guest 内 overlayfs pivot_root）。恢复零成本。
- **数据盘（10-20GB 主体）**：ZFS dataset 上的 raw file，从模板/上次快照 `zfs clone`（毫秒级），挂载瞬时、数据按需读。调优：`recordsize=16k~64k`、`primarycache=metadata`。不用 zvol（实测慢 25%~6 倍且有快照空间陷阱，《04》§1）。

**快照对象（pause 时产出）：**

| 对象 | 内容 | 生成方式 |
|---|---|---|
| memfile diff | 脏内存页（chunk 化 + lz4 + 零页跳过） | dirty-page bitmap 只写脏页 |
| WS 文件 | 工作集页访问 trace（≈内存 9%） | 首次 resume 记录，滚动更新 |
| disk diff | 数据盘块级增量 | `zfs send -i` |
| snapfile | vCPU/设备状态（KB 级） | Firecracker API |
| manifest | chunk 索引、来源层、校验、`(fc_version, kernel_version)` 标签 | 自研元数据 |

**chunk 仓库（L1/L2）：** FastCDC 内容寻址 → 同 base 派生沙箱间去重（60-80%+，去重域限租户内 + 公共 base 层）→ Garage/SeaweedFS 或 Hetzner Object Storage → lifecycle 转冷层。演进方向为 Xet 式 Merkle + block 聚合 + synthetic full（《04》#6）。

## 3. 沙箱生命周期状态机

```
 CREATE ──▶ RUNNING ──(idle timeout / pause)──▶ PAUSED-HOT (本机 NVMe，快照写穿 L1)
              ▲                                     │ (N 小时未唤醒, 释放本机)
              │                                 PAUSED-WARM (L1 对象存储)
              │ resume                              │ (D 天未唤醒, lifecycle 转储)
              │                                 ARCHIVED-COLD (低价冷层)
              │                                     │ (R 天未唤醒, 用户确认/策略)
              └── (选择性恢复重要文件) ◀──────── RECYCLED (仅保留 artifacts)
```

**三级恢复 SLO**（延迟预算拆解见《04》§2）：

| 状态 | 数据位置 | 恢复路径 | 目标延迟 |
|---|---|---|---|
| PAUSED-HOT | 本机 NVMe ZFS | zfs clone + uffd + WS 预取 | **P50 < 500ms** |
| PAUSED-WARM | L1 对象存储 | WS chunk 并行拉取（≥25GbE）+ 后台补全 | **P99 < 3s 可交互** |
| ARCHIVED-COLD | 冷层 | 毫秒级取回 + 热点 chunk 优先回填 | < 10s 可交互 |

关键机制：placement 粘性（优先回原节点命中 L0）；WS prefetch（上次访问 trace 优先预取）；pause 优化（balloon inflate + Free Page Reporting，CodeSandbox 实证 16GiB 快照 13GiB 可跳过）；**pause 后快照立即写穿 L1**（宿主故障不丢数据，RPO ≤ 5min）；唤醒直方图 pre-warm（预测唤醒前拉回本地并预解压 WS）。

## 4. 网络设计

- 每沙箱独立 netns + TAP + veth，NAT（MASQUERADE/DNAT）——解决快照 clone 后 guest IP 相同问题；netns 预建池化（~500 个后创建速率骤降）
- 沙箱流量 `--notrack` 防 conntrack 耗尽；FC 网络/块设备 rate limiter 按租户设置
- resume 后 guest 侧 flush ARP、chrony 校时（或 kvm-clock `clock_realtime`）
- 入向：Gateway 按 `sandboxID` 路由到 worker DNAT 端口
- 出向（zero-trust egress）：egress 代理统一出口——网络策略、域名白名单、凭证注入（token 不进沙箱）、secret redaction

## 5. 安全设计

- jailer：chroot + pivot_root + 每 VM 独立 uid/gid + `--new-pid-ns` + cgroup v2；默认 seccomp
- 宿主侧信道加固全集见《04》§5（nosmt/l1tf/禁 KSM/禁 swap 等）
- 快照唯一性：VMGenID（内核 reseed PRNG）+ guestd resume hook 刷新应用层 token；默认"一份快照 resume 一次"，fork 显式声明
- 快照静态加密（L1/L2 SSE 或客户端加密），manifest 带校验和（FC 本身只做 CRC）
- 多租户：沙箱间无东西向网络；配额与限速在 API 层

## 6. 技术选型汇总

| 决策点 | 选型 | 理由（数据依据见《04》§1） |
|---|---|---|
| 隔离运行时 | Firecracker（KVM） | 行业统一答案：硬件隔离 + 原生 snapshot/diff/uffd；E2B/Cursor/Fly/Lambda 同选 |
| 备选运行时 | Kata/gVisor 不采用；Docker 仅限受信任务 | Manus 实测 Docker spawn 10-20s 且无完整 OS 能力 |
| 本地存储 | ZFS dataset + raw file（NVMe 池） | clone O(1)、send -i 增量、lz4 压缩免费；zvol 已被实测数据否决 |
| rootfs | squashfs 只读 base + overlayfs 可写层 | E2B 验证的省盘方案；演进 composefs（《04》#7） |
| virtio-blk io_engine | Sync（默认） | Async 仍 dev preview 且拖累 restore +110ms |
| 内存恢复 | uffd + WS 预取（必选）+ chunk 化 lz4 | 纯 lazy 43MB/s 陷阱 vs 预取 533MB/s；CodeSandbox 实证 avg 400ms |
| 对象存储 | Garage/SeaweedFS 自建，或 Hetzner Object Storage | MinIO 2026-04 已归档停更；Hetzner OS 同区取回免费 |
| 冷归档 | B2/R2 IA/Hetzner OS；Glacier IR 仅限极少唤醒 | Glacier IR 取回 $0.03/GB，唤醒 >7 次/年不划算 |
| 分布式 FS（可选阶段三） | JuiceFS 挂 /workspace | 跨机恢复友好；冷 29ms/热 0.3ms 需缓存兜底 |
| Ceph RBD | 暂不采用 | O(1) clone 最优但运维重，留作规模化选项 |
| 元数据/编排 | PostgreSQL + Redis；后期 Temporal | E2B/Cursor 同构；不自研状态同步 |
| 硬件 | Hetzner/OVH 裸金属 + NVMe + ≥25GbE 内网 | 成本表见《03》§5；25G 是温恢复 SLO 的硬前提 |
| 参考代码 | e2b-dev/infra 深度复用 | 唯一生产级开源实现；Apache 部分可并入（AGPL 兼容），BSL 部分只读不抄 |

## 7. 与 E2B 开源版的差异点（EmberVM 自研部分）

1. **大数据盘（10-20GB）一等公民**：独立数据盘 + `zfs send -i` 增量归档（E2B 文件系统上限 1-5GB）
2. **分层冷归档**：HOT/WARM/COLD/RECYCLED 四级自动流转（E2B 只有 pause 一级）
3. **WS 预取 + 调度粘性 + 唤醒预测**：面向"休眠-唤醒"高频场景
4. **零依赖自托管**：无 Nomad/Consul/GCP，单一 Scheduler + Node Agent 心跳（E2B self-host 官方承认未解决网络隔离/LB/部署三件事）
5. **fork/branch Agent 原语**（M5，《04》#4）：E2B 开源版没有
