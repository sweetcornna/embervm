## 1. 设计决策验证表（VERDICT）

| 决策 | 判定 | 依据 |
|---|---|---|
| ZFS zvol 做沙箱数据盘 | ❌ **推翻 → ZFS dataset 上的 raw file** | NVMe 上 zvol 仅达盘面 IOPS ~1/2（openzfs#17032）；raw-file-on-dataset 快 25%~6 倍（Klara/Proxmox 实测）；zvol 有快照空间陷阱（refreservation）与写停顿 issue（#11407）。零个已知生产 Firecracker 平台用 zvol：Fly.io=LVM thin、E2B=ext4+NBD CoW、CodeSandbox=XFS reflink、Depot/Blacksmith=本地 NVMe Ceph。dataset+raw file 保留 send/recv、压缩、clone 全部能力且更快；调优 `recordsize=16k~64k`、`primarycache=metadata` |
| Firecracker io_engine | ✅ Sync（默认） | Async(io_uring) 至今 dev preview（PID 耗尽/cgroup 问题阻塞 GA），且设备创建 +110ms 拖累 restore。跟进：v1.13+ PCIe transport（`--enable-pci`） |
| uffd 懒加载内存恢复 | ✅ 确认，但 **WS 预取为必选项** | 纯逐页 lazy fault 只能榨出 SSD 的 43MB/s（REAP 实测）；预取/批量后 260-533MB/s。REAP：工作集平均仅占内存足迹 9%、97% 页跨调用复现。Cloud Hypervisor 对照：eager 7.1s vs uffd 83ms（2GB） |
| chunk 压缩算法 | ✅ 热路径 lz4（8-16KiB chunk），归档层 zstd | lz4 解压 ~5GB/s/核 vs zstd ~1.4-2GB/s；CodeSandbox 试过 4KiB-2MiB 定型 8KiB lz4；zstd+字典解压也只追平 lz4——**字典仅用于归档层** |
| 密度 50 沙箱/节点 | ✅ 确认（保守） | VMM 开销 ~3MB/VM；Lambda 生产数百至数千 VM/机、内存超卖生产 10x；官方 netns 方案 100 clones 实测无恙（ping 0.315ms、per-VM 2.25Gbps） |
| MinIO 自建对象存储 | ❌ **推翻 → Garage / SeaweedFS / Hetzner Object Storage** | minio/minio 仓库 2026-04-25 正式 archived，无安全补丁。自建选 Garage v2.3（注意无 versioning/object-lock）或 SeaweedFS（S3 API 更全，含 lifecycle）；省事用 Hetzner OS（Ceph 底座，≈€4.89/TB·月，S3 操作免费） |
| 冷归档用 Glacier IR | ⚠️ 仅限"极少唤醒"沙箱 | 取回费 $0.03/GB——20GB 沙箱唤醒一次 = $0.60，唤醒 >7 次/年即不如 Hetzner OS/B2（无取回费）。见 §7 |
| Hetzner 25GbE 内网 | ✅ 成立，但不是 vSwitch | vSwitch 复用 1G uplink（陷阱）。正解：私有 25G 交换机 €68/月 + 每台 Dual NIC €12 + LAN 25G €7.5，3 节点净增 ≈€127/月；OVH vRack 25G 为产品化标配免施工。成本表见《03》§5 |
| 16 周到生产 | ⚠️ 校准：16 周内部 MVP，4-6 个月可收费 beta | E2B 官方承认 self-host 未解决的三件事（沙箱网络隔离、LB、部署工具）正是 demo→生产的主要工作量 |

## 2. 端到端延迟预算模型（10-20GB 沙箱，4-8GB RAM）

**热恢复（PAUSED-HOT，数据在本机 NVMe）—— 目标 P50 < 500ms**

| 步骤 | 预算 | 依据 |
|---|---|---|
| snapfile 加载 + 设备重建 | 5-20ms | 官方基线 512MB <8ms；netns 预建池化 |
| WS 文件顺序读 + lz4 解压（WS≈内存的 9%，8GB→~700MB，压缩后 ~300MB） | 100-300ms | REAP 533MB/s（O_DIRECT）+ lz4 5GB/s/核 |
| 数据盘 ZFS clone + 挂载 | <10ms | O(1) 元数据 |
| vCPU resume + guest 校时/ARP flush | 10-50ms | |
| 剩余内存页后台 uffd 回填 | 不阻塞（FaaSnap concurrent paging） | 缺页兜底 24.6µs/read ≈ 260MiB/s |

参照：CodeSandbox 生产 avg 400ms / P99 2s（250 万次/月）——模型与实证吻合。

**温恢复（PAUSED-WARM，对象存储 + 25GbE + 本地缓存）—— 目标 P99 < 3s 可交互**：WS chunk（~300MB 压缩）并行拉取 @2-3GB/s ≈ 150-300ms，其余同热路径；非 WS 页与数据盘冷块后台回填（15GB 全量补齐 ≈ 1-2 分钟，不阻塞交互）。**内网只有 1-10GbE 时此目标不成立——采购必须锁定 ≥25G。**

**冷恢复（ARCHIVED-COLD）—— 目标 <10s 可交互**：Hetzner OS/B2/R2 取回首字节均毫秒级，瓶颈仍是 WS 拉取带宽。

## 3. 前沿技术地图

### 3.1 学术界快照恢复研究

| 工作 | 核心思想 | 数据 |
|---|---|---|
| **REAP** (ASPLOS'21) | 首次 resume 记录页访问 trace → 紧凑 working-set 文件 → 后续恢复一次顺序读 eager 预取 | 消除 97% 缺页，冷启动加速 3.7x，纯 userspace（uffd） |
| **FaaSnap** (EuroSys'22) | concurrent paging：VM 立即恢复执行，working set 后台并发装载 | 端到端最高 3.5x，平均只比"快照常驻内存"慢 3.5%；代码开源 |
| **Catalyzer** (ASPLOS'20) | sfork：从运行中的沙箱直接 fork 而非从文件 restore | 最佳 <1ms；fork 比 restore 快一个量级 |
| **Sabre** (OSDI'24) | Intel IAA 硬件压缩/解压快照页 | 压缩 4.5x，解压开销近零，恢复提速 55%；已集成 Firecracker |
| **DeltaBox** (2026) | 面向 stateful AI agent：DeltaFS（冻结层+切层回滚）+ 增量 dump + 模板进程 fork 恢复 | checkpoint 14ms、rollback 5ms（SWE-bench/RL）；与本项目场景同构 |
| **TrEnv** (SOSP'24) | mm-template：内存状态放 CXL/RDMA 共享池跨节点复用；serverless 函数 24-90% 内存访问是只读页 | 多机共享内存模板最前沿（远期参考） |
| **On-demand-fork** (EuroSys'21) | CoW 推广到页表本身 | 1GB 进程 fork 快 65x（未合入主线，暂不采用） |
| **Serverless in the Wild** (ATC'20) | 唤醒间隔直方图 → pre-warm + keep-alive window；不可判别时 ARIMA | 更少资源大幅减少冷启动 |

### 3.2 关键反结论（少走弯路）

- **KSM 内存去重：多租户禁用**——CVE-2024-0564 实锤跨租户侧信道，Proxmox 官方建议关闭，且扫描烧 CPU；安全替代见创新点 #3
- **CRIU 容器 checkpoint 不做主路径**（TCP/GPU/挂载限制多）
- **lazy loading 对"启动后立刻顺序读大文件"的负载可能更慢**（如加载模型权重）→ 必须配 WS 预取兜底
- **GPU checkpoint 未成熟**：cuda-checkpoint/CRIUgpu 单卡推理 alpha 可用（Modal 实测 7-13B 模型 60-90s→2-5s），UVM/NCCL 不支持 → 实验特性

## 4. 十二个创新点（影响 × 可行性排序）

### 第一梯队：秒恢复底座（M2 落地）

**#1 压缩快照 + uffd 按 chunk 惰性解压** ⭐⭐⭐⭐⭐/⭐⭐⭐⭐⭐
8-16KiB chunk 粒度 lz4 压缩 memfile（热路径锁定 lz4，见 §1），带偏移索引随机读；缺页按 chunk 在线解压；按页选择性压缩（可压缩才压）。CodeSandbox 实证：8GiB→745MiB（~10x），resume 4s→1s。**20GB 状态压到 2-3GB，是"网络秒级恢复"的第一杠杆。**

**#2 Working-set record + prefetch（REAP + FaaSnap 合体，必选项）** ⭐⭐⭐⭐⭐/⭐⭐⭐⭐
首次 resume 记录页访问 trace → WS 文件（≈内存足迹 9%）→ 后续恢复一次顺序读装载 WS（消除 97% 缺页），其余页后台并发装载，VM 不等加载完即执行。与 #1 叠加：WS 优先预取解压。**学术界验证、竞品尚未普遍产品化的空档；不做它，uffd 懒加载本身就是性能陷阱（43 vs 533MB/s）。**

**#3 模板 memfd 共享 + uffd minor faults（拒绝 KSM 的安全 dedup）** ⭐⭐⭐⭐⭐/⭐⭐⭐⭐
同模板 N 个沙箱共享同一份 page cache/memfd 模板：clean 页全宿主共享，写入页 CoW 私有化；进阶用 uffd minor faults（Linux 5.13+，页在 page cache 只缺 PTE，handler 直接 CONTINUE）。只在"自己的模板"内共享，无跨租户侧信道，内存密度提升数倍。

### 第二梯队：产品差异化（M3-M5 落地）

**#4 VM fork/branch 作为一等 API（Agent 原生原语）** ⭐⭐⭐⭐⭐/⭐⭐⭐
snapshot/branch/rollback 暴露给 Agent：uffd CoW 内存克隆（CodeSandbox 实证任意内存大小 <1.5s fork）+ ZFS/overlayfs 层链磁盘（checkpoint=冻结层，rollback=切层）。DeltaBox 证明 checkpoint 14ms / rollback 5ms 可达。场景：tree-of-thought 并行探索、RL rollout fan-out、time-travel 调试（每步自动快照 + 任意步 fork 重放）。对标 Morph Infinibranch，E2B 开源版没有。

**#5 直方图预热预测（唤醒预测 × 分层存储的乘法效应）** ⭐⭐⭐⭐/⭐⭐⭐⭐⭐
移植 Azure《Serverless in the Wild》：每沙箱唤醒间隔直方图 → pre-warm window，预测唤醒前把快照从冷层拉回本地 NVMe 并预解压 WS；CV 过低回退固定 keep-alive，数据不足用 ARIMA。约一周工作量，直接砍温/冷恢复 P99。分层越深，预测越值钱；直方图同时用于选择归档目的地（见 §7）。

**#6 Xet 式 chunk 仓库（Merkle + block 聚合 + synthetic full）** ⭐⭐⭐⭐/⭐⭐⭐⭐
借鉴 HuggingFace Xet（~21PB，协议公开）：8-128KB CDC chunk + Blake3 Merkle 树 + 三级去重 + chunk 聚合 ~64MB block 降请求数。配合 forever-incremental + synthetic full（Veeam 范式）：周期性元数据级 synthetic full（CDC 去重下几乎零搬移），冷恢复永远只读"一层"。

### 第三梯队：密度与成本（M4 及以后）

**#7 EROFS file-backed mount + composefs 共享 rootfs** ⭐⭐⭐/⭐⭐⭐⭐ —— Linux 6.12 起千沙箱共享不可变镜像免虚拟块设备；内容寻址 + fs-verity；rootfs 彻底移出快照体积。
**#8 三档恢复统一为"层切换"语义（DeltaBox）** ⭐⭐⭐⭐/⭐⭐⭐ —— 热=本机层切换（ms）、温=快照 clone（百 ms）、冷=远端回填（秒）；rollback 与 resume 共用机制，架构收敛。
**#9 ublk 惰性块设备（zero-copy）** ⭐⭐⭐/⭐⭐⭐ —— 数据盘冷回填走 ublk（io_uring，6.0+）按需拉块；`UBLK_F_AUTO_BUF_REG` 把 daemon CPU 从 ~10% 降到 ~1.5%。
**#10 MGLRU/DAMON 热度感知休眠** ⭐⭐⭐/⭐⭐⭐ —— 内核工作集估计决定休眠页压缩/下沉顺序；checkpoint 按热度排序写出 = 恢复顺序。
**#11 GPU 沙箱 checkpoint（实验卡位）** ⭐⭐⭐⭐(远期)/⭐⭐ —— cuda-checkpoint + CRIUgpu（已进 CRIU 主线）feature flag 接入，等 NCCL/UVM 支持后转正。
**#12 VectorCDC / Sabre IAA 硬件加速** ⭐⭐/⭐⭐⭐ —— SIMD chunking 快 15-207x；IAA 仅 Sapphire Rapids+；规模化后再做。

## 5. 宿主机加固与调优清单（生产必做）

**安全（多租户红线）**
- jailer 强制：chroot + pivot_root + 每 VM 独立 uid/gid + `--new-pid-ns` + `--cgroup-version 2`；默认 seccomp（全线程仅 ~50 syscalls）
- 侧信道：`nosmt=force`、`l1tf=full,force`、KPTI 验证 `Mitigation: PTI`、`spec_store_bypass_disable=on`、及早加载 microcode、spectre-meltdown-checker 验证
- 禁 KSM、禁 swap、`kernel.unprivileged_bpf_disabled=1`
- host iptables 挡 IMDS：`-i tap+ -d 169.254.169.254 -j DROP`
- host mount point 多会拖慢 jail 创建（500 mounts + 10 并发 → 10 倍慢）

**性能**
- host kernel 6.1 LTS（官方支持 5.10/6.1/6.18）；host cmdline `quiet loglevel=1`（否则 aarch64 restore 3ms→8.5ms）
- guest：官方 microvm kernel config，boot args `reboot=k panic=1 8250.nr_uarts=0 swiotlb=noforce`
- cgroups v2（v1 下 restore 显著劣化）；kernel 6.1 需 `favordynmods` 或 `kvm.nx_huge_pages=never`
- hugepages 谨慎：boot 提速最多 50%，但快照只能经 UFFD 恢复且 dirty tracking 抵消收益
- ZFS：`recordsize` 16-64k 试验、`primarycache=metadata`、`sync=disabled` 仅限可重建的快照缓存盘（崩溃丢 ~5s）

**网络数据面**
- per-VM netns + veth + NAT（E2B 同款），netns 预建池化（~500 个后创建速率从 250/s 掉到 50/s）
- TAP 单 queue ~25Gbps、加成延迟 ~0.06ms；备选 tc-redirect-tap
- conntrack 默认 262144 条（~328B/条）；沙箱流量 `--notrack`——USENIX Sec'23 实证失控沙箱可致邻居 60% 丢包
- 避免大规模 Linux bridge（1024 口 + 广播放大）；FC 网络/块设备 rate limiter 按租户设置

**存储与磨损**
- 快照高频写 = NVMe 磨损（4GB VM 高频 pause 每天数百 GB）→ DC 级 NVMe、SMART TBW 监控、系统盘 RAID1
- `vm.dirty_bytes` 调低 + cgroup io.max 防脏页邻居干扰（论文实证 93.4% IO 降级）

## 6. 可靠性工程（来自生产事故复盘）

**SLO 设计**（对齐行业：Fly/E2B 均不承诺运行态存活）：API 可用性 99.9%；创建成功率 99.5%；热恢复 P95 < 2s；不承诺宿主故障下运行态存活，承诺持久数据 **RPO ≤ 5min**（pause 写穿 L1 + 运行中定期增量）。

**Fly.io 两个血泪教训**
1. 本地 NVMe 卷 + 宿主故障 = 数据永久丢失（社区多起事故）→ pause 后立即异步上传 L1（写穿），本机只作缓存——不可妥协项
2. Corrosion（自研 gossip/CRDT 状态同步）一次 schema 变更全网瘫痪 11.5h → 控制面用单一 PostgreSQL 权威状态 + 简单心跳，绝不自研分布式状态同步

**快照版本兼容 SOP**
- Firecracker：patch 内格式稳定、backward 尽力、1.6+ 无 forward 兼容；跨宿主内核版本恢复官方定义 unstable
- 每个快照打 `(fc_version, host_kernel_version)` 标签，调度限同构池
- 升级 = 蓝绿节点：新节点跑新版 → drain → 旧快照在旧环境唤醒后用新版重拍，或容忍冷启动重建；绝不原地双升级后 restore 旧快照

**控制面 PostgreSQL**：2-3 人团队不上 Patroni；托管 PG 或 单主 + streaming replica + WAL-G 归档（RPO 秒级、RTO 10-30min）；控制面挂掉时数据面沙箱本地续租继续存活。

**uffd handler 单点**：每 VM 独立 handler 进程 + systemd 看护 + 缺页超时告警；handler 挂 = 缺页永久卡死，watchdog 强制回收并从最近快照重建。

**迁移**：Firecracker 不支持 live migration；跨机"温迁移"唯一成熟路径 = chunk 化快照 + 异机 uffd restore（BuildBuddy 生产验证：UFFD 管内存 + FUSE 管磁盘 + 本地 NVMe chunk 缓存 + 远端 cache）。

## 9. 数据缺口（诚实标注，列入 M0 实测）

官方无 2/4/8/16GB restore 延迟表（仅 512MB <8ms 基线）；UFFDIO_COPY 多线程吞吐无权威表；`sync=disabled` NVMe 加速无权威百分比；E2B 对象存储 chunk 拉取延迟未公开。

## 10. 创新点特有风险

| 风险 | 应对 |
|---|---|
| uffd minor + guest_memfd 支持仍在演进（2025-26 内核补丁） | MVP 用成熟的 MAP_PRIVATE 共享路线，minor faults 作增量优化 |
| WS trace 对负载漂移敏感 | FaaSnap 式后台并发装载兜底；trace 滚动更新 |
| fork 分支共享熵/token（#4 放大"同快照多次恢复"问题） | 每分支 VMGenID 更新 + guestd resume hook 强制刷新；API 文档明示语义 |
| Xet 式全局去重的跨租户侧信道（chunk 存在性） | 去重域限定租户内；全局去重仅对公共 base 层 |
