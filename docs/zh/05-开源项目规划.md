# 开源项目规划（GitHub）

> 日期：2026-07-03。本项目定位为 **GitHub 开源软件服务**：任何人可在自己的云服务器/裸金属上部署一套沙箱云。本文确定命名、许可、仓库结构、社区与发布策略。

---

## 1. 项目命名

工作名（推荐）：**EmberVM**（余烬）

- 隐喻：Firecracker（鞭炮）炸响之后留下 ember（余烬）——**看似熄灭，吹一口气瞬间复燃**。精准对应产品核心：休眠沙箱秒级唤醒 + 热/温/冷分层。
- 与 Firecracker 生态同一意象体系，社区易联想；中文名"余烬"简洁有记忆点。
- Tagline：*The sandbox cloud that never goes cold.*（永不冷透的沙箱云）

备选：Thawbox（解冻+沙箱）、Cryobox（冷冻仓）、Permabox（持久沙箱）、Sandwarm（暖沙箱/沙虫谐音）。

⚠️ 注册 GitHub org / 域名前需检索名称可用性与商标冲突（embervm 与 Ember.js 属不同领域，风险低但需确认）。本地文件夹 `dynamic_docker` 可在建仓时更名。

## 2. 许可证：AGPL-3.0（已决策）

- **动机**：本项目是网络服务软件，Apache-2.0 下云厂商可闭源托管盈利而不回馈；AGPL-3.0 的网络条款要求"修改后对外提供服务也必须开源"，保护社区成果。Daytona（同类沙箱云）同选 AGPL-3.0 且全功能开源。
- **商业化预留**：所有外部贡献者签 **CLA**（DCO + CLA 双轨），项目方保留双许可（dual-license）能力——未来可向企业出售商业许可/托管版而不受 AGPL 约束。
- **代码复用合规红线**：
  - Apache-2.0 代码（如 e2b-dev/infra 的 Apache 部分、firecracker-microvm）**可以**纳入 AGPL 项目（单向兼容），保留原 NOTICE/版权头；
  - e2b-dev/infra 中 **BSL 授权的部分不可复制**——只读其设计、自行实现；
  - 每次引入第三方代码在 `THIRD_PARTY_NOTICES.md` 登记。

## 3. 仓库结构（规划，本轮不落码）

```
embervm/
├── README.md / README.zh-CN.md      # 英文为主，中文对照
├── LICENSE (AGPL-3.0)  CONTRIBUTING.md  CODE_OF_CONDUCT.md  SECURITY.md
├── THIRD_PARTY_NOTICES.md
├── cmd/                             # Go 入口
│   ├── apiserver/                   # 控制面 API
│   ├── nodeagent/                   # 节点守护（FC/netns/ZFS/快照，需 root）
│   ├── uffd-handler/                # 内存懒加载 handler（每 VM 一进程）
│   └── guestd/                      # VM 内 daemon
├── pkg/                             # scheduler / lifecycle / chunkstore / template ...
├── deploy/
│   ├── singlenode/                  # 单机一键部署（云服务器实测入口）
│   ├── ansible/                     # 多节点生产部署
│   └── terraform/                   # Hetzner/OVH/GCP 供应
├── docs/
│   ├── (en)                         # 英文文档（随代码增量补齐）
│   └── zh/                          # 中文全套（本套 01-06）
├── test/
│   ├── integration/                 # CI 内真 microVM 测试（GitHub Actions 有 KVM）
│   └── bench/                       # M0 基线基准套件（恢复延迟/吞吐矩阵）
└── .github/workflows/               # CI（见《06》§4）
```

## 4. 版本与发布

- SemVer；`v0.x` 阶段 API 可破坏但需 CHANGELOG 声明；**快照格式独立版本号**（吸取 Firecracker 教训：manifest 内嵌 `snapshot_format_version` + `(fc_version, kernel_version)` 标签，从 v0.1 就做）
- 发布物：GitHub Releases 二进制（linux/amd64 优先；arm64 跟随 Firecracker 支持）+ 一键安装脚本 `deploy/singlenode/install.sh`
- 与上游对齐：跟踪 Firecracker minor 版本（当前 v1.16.x），在 CI 矩阵中同时测 N 和 N-1

## 5. 社区与治理

- **CONTRIBUTING.md**：DCO+CLA、conventional commits、good-first-issue 标签体系；架构决策用 ADR（`docs/adr/`，本套 01-04 文档的结论沉淀为 ADR-0001..N）
- **SECURITY.md**：多租户隔离是本项目的安全命脉——私密披露渠道 + 48h 响应承诺；沙箱逃逸类漏洞最高优先级
- **路线图公开**：《03》里程碑 M0-M5 映射为 GitHub Milestones；核心指标（恢复 P50/P99）做成公开的基准看板（bench 结果 CI 自动发布到 gh-pages）
- **定位声明**（README 首屏）：vs E2B（自托管友好度：我们生来为自托管设计，无云厂商耦合）、vs Daytona（隔离强度：Firecracker microVM 硬隔离 vs 默认容器）、独有：10-20GB 大数据盘一等公民 + 分层冷归档 + WS 预取秒级恢复

## 6. 开源特有的架构约束（对《02》的增补）

1. **零云厂商耦合**：对象存储走 S3 兼容接口（Garage/SeaweedFS/任意 S3）、元数据只依赖 PostgreSQL+Redis、部署不依赖 Nomad/Consul——这正是 E2B self-host 最被诟病处（GCP+Nomad 全家桶），是本项目的差异化机会
2. **单机模式为一等公民**：`embervm dev` 单进程模式（API+scheduler+nodeagent 合体），一台便宜云服务器即可完整体验——降低贡献者门槛的关键（详见《06》）
3. **可测试性**：所有组件在无 KVM 环境可 mock 运行（单元测试）；真 microVM 测试在 CI 的 KVM runner 上跑（GitHub Actions ubuntu-latest 自 2024-01 起有 /dev/kvm，免费公共仓库可用——已验证，见《06》§4）
